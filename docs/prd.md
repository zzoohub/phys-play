## 1. Context

문제 정의, 페르소나, 시장 분석, 가정/리스크는 [Product Brief](./product-brief.md)를 참조한다.

### Elevator Pitch

과학 개념을 **"예측하고, 실험하고, 발견하는" 3D 게임**으로 만든다. 사용자는 매 챌린지에서 결과를 먼저 예측하고, 3D 시뮬레이션에서 God Hand(1인칭 테이블탑 조작)로 직접 실험한 뒤, 예측과 결과를 비교하며 과학 원리를 스스로 발견한다. 시뮬레이션 엔진 기반의 무한 챌린지 생성과 AI 적응형 난이도 조절로, 틀린 예측이 다음 단계의 호기심이 되는 자기강화 루프를 만든다.

### Mental Model

**"나만의 과학 연구소"** — 사용자는 자신의 연구소에서 실험자로서 과학을 탐험한다. 연구소에는 여러 공간(역학 실험실, 분자 실험실, 우주 관측소, 양자 연구소)이 있고, 각 공간에는 여러 스테이션(실험대)이 있다. 스테이션에서는 무한한 챌린지가 생성되며, 챌린지를 클리어하면 새로운 스테이션과 공간이 해금된다. 게임을 할수록 내 연구소가 넓어지고 깊어진다.

### Core Value Propositions

| Value Prop | 해결하는 문제 |
|-----------|-------------|
| **예측-실험-발견 코어 루프** | 수동성 — "생각하지 않고 만지는" 기존 도구와 달리, 인지적 갈등을 핵심 메카닉으로 설계 |
| **3D-native 시뮬레이션** | 차원 — 3D 개념을 3D로 직접 보고, 만지고, 실험 |
| **직관 우선, 수식은 나중** | 순서 — 먼저 체험으로 직관을 잡고, 수식은 이해의 도구로 나중에 등장 |

## 2. Core Experience Design

### 2.1 Core Loop: 예측 → 실험 → 발견

PhysPlay의 모든 경험은 하나의 루프로 수렴한다.

```
┌─────────────────────────────────────────────┐
│                                             │
│   🎯 PREDICT                                │
│   "이 공을 달에서 던지면 어디에 떨어질까?"      │
│   → 사용자가 궤적을 그리거나 선택지를 고른다    │
│                                             │
│           ↓                                 │
│                                             │
│   🧪 PLAY                                   │
│   내 앞의 실험 대상을 직접 손으로 조작한다      │
│   웹: 마우스/터치 | XR: 핸드 트래킹            │
│                                             │
│           ↓                                 │
│                                             │
│   💡 DISCOVER                                │
│   예측 vs 결과 비교                           │
│   ✗ 틀렸다면 → "왜?" → 개념 설명 + 변수 조절  │
│   ✓ 맞았다면 → 변수 추가 → 더 어려운 예측     │
│                                             │
│           ↓                                 │
│                                             │
│   🔄 NEXT CHALLENGE                          │
│   AI가 수준에 맞는 다음 챌린지를 제시           │
│   호기심이 다음 단계의 동기가 된다              │
│                                             │
└─────────────────────────────────────────────┘
```

**코어 루프가 학습 동기를 만드는 원리:**

1. **예측은 참여를 강제한다** — 슬라이더를 만지작거리는 것과 달리, "내 예측"이 걸려 있으면 결과에 관심을 갖게 된다
2. **틀림은 호기심을 만든다** — "왜 내 예상과 다르지?"라는 질문이 자연스럽게 개념 학습으로 이어진다
3. **맞음은 자신감을 만든다** — 더 어려운 챌린지에 도전하는 동기가 된다
4. **연결은 탐험을 만든다** — "달에서 맞았으니, 블랙홀 근처에서는?"이 트랙 간 이동의 동기가 된다

### 2.2 Prediction Types (예측 방식)

예측 방식은 챌린지 성격에 따라 달라진다. 챌린지 데이터에 `predict.type`으로 정의되어 UGC에서도 유저가 예측 방식을 선택할 수 있다.

| 예측 유형 | 설명 | 적합한 챌린지 | 정답 판정 |
|----------|------|-------------|----------|
| **궤적 그리기 (trajectory)** | 사용자가 화면에 궤적을 직접 그린다. "내가 그린 선 vs 실제 궤적"이 겹쳐 보이는 순간이 가장 강력한 인지적 갈등 | 투사체 운동, 행성 궤도, 삼체 문제 | 오차 범위 기반 |
| **이진 선택 (binary)** | "될까 / 안 될까", "이쪽 / 저쪽" 같은 양자택일 | 터널링 투과, 충돌 방향, 공명 발생 | 정답/오답 |
| **패턴 선택지 (pattern)** | 여러 결과 패턴 중 하나를 고른다 | 간섭 패턴, 분자 대칭, 오비탈 모양 | 선택지 매칭 |
| **3D 배치 (placement)** | 3D 공간에 마커, 오브젝트, 구조물을 직접 배치한다. "내가 놓은 위치 vs 정답 위치"를 비교 | 분자 조립, 라그랑주 점 찾기, 안티 스피커 배치, 악기 타격점 | 위치/방향 오차 기반 |

### 2.3 Screen Composition: 풀스크린 3D + HUD

실험실에 진입하면 화면 전체가 3D 뷰포트다. Predict/Discover 텍스트, 버튼, 설정 등 **모든 UI는 3D 위에 떠 있는 HUD(Head-Up Display) 오버레이**로 표시한다. 2D 페이지로 빠지지 않으므로 몰입이 끊기지 않는다.

| 레이어 | 해당 영역 | 설명 |
|--------|----------|------|
| **3D Viewport (풀스크린)** | 실험실 내 모든 단계 | God Hand 조작, 시뮬레이션 실행, 결과 시각화. 실험실 진입 후 화면 전체를 차지 |
| **HUD Overlay** | Predict, Discover, 실험 설정 | 질문/입력 UI, 설명 패널, 버튼, 변수 조절 등. 3D 뷰포트 위에 반투명 오버레이로 표시 |
| **2D Page** | 허브, 설정, 진도 | 연구소 허브(공간/스테이션 선택), 진도, 설정 등 실험실 바깥 화면만 2D로 처리하여 리소스 절약 |

이 구조의 장점: (1) 코어 루프(예측→실험→발견) 전체가 3D 환경 안에서 완결되어 몰입이 끊기지 않는다. (2) HUD는 게임에서 검증된 UI 패턴이라 사용자에게 자연스럽다. (3) XR 전환 시 HUD가 공간 고정 패널로 자연스럽게 매핑된다.

### 2.4 Interaction Model: 1인칭 실험자 + 테이블탑 조작 (God Hand)

캐릭터나 아바타 없이, **"나는 실험자이고, 실험 대상은 내 앞에 있다"**는 일관된 프레임을 모든 모듈에서 유지한다. 이를 "신의 손(God Hand)" 또는 테이블탑 모델이라 부른다.

**왜 이 모델인가:**

- **스케일 전환이 자유롭다** — 태양계를 테이블 위에 축소해서 올려놓고 손으로 돌릴 수도, 분자를 눈앞에 확대해서 손으로 조립할 수도 있다. 원자든 은하든, "내 앞에 놓인 것을 관찰하고 조작한다"는 프레임이면 자연스럽다
- **코어 루프와 완벽하게 맞는다** — "이 공을 던지면 어디에 떨어질까?" → 손으로 직접 던진다 → 결과를 관찰한다. 전체 과정이 "내 손 → 내 눈" 안에서 완결된다
- **웹 ↔ XR 전환 비용이 제로다** — 웹에서 마우스로 시뮬레이션을 조작하는 것이 이미 테이블탑 위의 대상을 조작하는 느낌이다. XR로 전환하면 마우스가 내 손으로 바뀌는 것뿐, 멘탈 모델이 동일하다
- **캐릭터 개발 부담이 없다** — 아바타, 커스터마이징, 3인칭 카메라 등의 복잡도를 완전히 제거한다

**디바이스별 인터랙션 매핑:**

| 디바이스 | 인풋 | God Hand 매핑 | 비고 |
|---------|------|-------------|------|
| **PC (Primary)** | 마우스/키보드 | 클릭 드래그로 던지기, 마우스 드로잉, 스크롤 줌, 키보드 단축키 | 넓은 화면 + 정밀 입력으로 최적 경험. 풀스크린 3D + HUD가 가장 잘 동작 |
| **모바일** | 터치 제스처 | 스와이프로 던지기, 손가락 드로잉, 핀치 줌 | 반응형 레이아웃 + 터치 제스처 매핑으로 핵심 경험 유지 |
| **XR** | 핸드 트래킹 | 직접 손으로 집기/던지기, 공중 드로잉, 손 제스처 줌 | 같은 엔진에서 입력만 매핑. 3D → XR 점진적 전환 |

**God Hand 인터랙션 패턴:**

God Hand는 단순 슬라이더 조작이 아닌, **물리적·체감적 조작**이다. 모든 챌린지의 God Hand는 아래 패턴의 조합으로 구성된다.

| 패턴 | 웹(3D) — 마우스/터치 | XR — 핸드 트래킹 | 예시 |
|------|---------------------|-----------------|------|
| **던지기/발사** | 클릭 드래그 방향/강도 | 직접 잡아 던진다 | 투사체, 행성 궤도 투입 |
| **조립/배치** | 드래그로 위치 설정, 스냅 결합 | 양손으로 잡아 맞춘다 | 분자 조립, 트랙 빌딩 |
| **당기기/밀기** | 드래그로 변형 | 손으로 당기고 민다 | 피스톤 압축, 진자 당기기 |
| **설치/제거** | 클릭으로 토글, 드래그로 배치 | 잡아서 놓기/빼기 | 검출기 설치, 촉매 배치 |
| **공간 흔들기** | 마우스 쉐이크 | 공간을 흔든다 | 온도 올리기(분자 운동) |

→ 각 엔진별 God Hand 상세는 [`docs/contents/engines/*/challenges.md`](./contents/engines/README.md) 참조

모든 경우에 **"나(실험자) → 내 앞의 대상 → 손으로 조작"**이라는 프레임이 일관된다.

### 2.5 Visual Design Direction

**Soft-Tech Educational** 기반 교육 플랫폼 + **Portal Transition**으로 진입하는 **Semi-Stylized 3D Sandbox**(God-Hand) 게이미피케이션 학습 환경.

| 레이어 | 톤 | 핵심 키워드 |
|--------|-----|-----------|
| 2D (Landing, Hub) | Clean & Airy | 밝고 가벼운 색감, 부드러운 라운드, 교육 플랫폼 신뢰감 |
| Portal Transition | Dimensional Shift | 2D→3D 경계를 극적으로 연출, 몰입 전환의 심리적 게이트 |
| 3D Lab | Immersive Lab | Semi-stylized 배경과 오브젝트, 게임 스테이지 몰입감 |

**실험실별 고유 분위기 — Semi-Stylized 3D 게임 스테이지처럼**
게이미피케이션의 핵심은 각 실험실이 교육 도구가 아니라 Semi-Stylized 3D 게임 스테이지처럼 느껴지는 것이다. 각 실험실(역학, 분자, 우주, 양자)은 고유한 비주얼 테마(조명, 색상 팔레트, 머티리얼, 파티클 이펙트)와 앰비언트 사운드를 가진다. 실험실을 해금하는 것 자체가 "새로운 스테이지를 여는" 흥분을 준다. 실험실 진입 시 전환 연출, 고유한 BGM, 시각적 리워드 이펙트 등으로 게임적 몰입을 극대화한다.

상세 비주얼 디자인 언어(컬러, 타이포, 모션 원칙 등)는 [UX Design](./ux-design.md)을 참조한다.

### 2.6 Onboarding: 첫 30초

**튜토리얼이 곧 첫 번째 챌린지다.**

**첫 방문 플로우:**

1. URL을 열면 회원가입/트랙 선택 없이 바로 역학 실험실의 "공 던지기" 챌린지로 진입
2. 30초 안에 예측 → 실험 → 발견 한 바퀴를 체험
3. 첫 챌린지 완료 후 연구소 허브가 열리며 전체 공간 맵이 등장
4. 해금된 공간과 잠긴 공간(실루엣)이 보여 호기심을 유발

**재방문 플로우:**

1. 연구소 허브(2D)가 홈 화면 — 실험실 바깥이므로 2D로 처리
2. 해금된 공간들이 보이고, 아직 잠긴 공간은 실루엣으로 표시
3. 공간 선택 시 전환 연출과 함께 풀스크린 3D + HUD 실험실로 진입
4. 실험실 내에서 스테이션/챌린지는 HUD로 선택 — 3D를 벗어나지 않음

## 3. Content Architecture

### 3.1 3개 레이어 구조

기존 "4트랙 × 6모듈 = 24개 고정 콘텐츠"의 교과서식 구조를 폐기하고, **시뮬레이션 엔진 기반의 확장 가능한 구조**를 채택한다.

#### Layer 1 — 시뮬레이션 엔진 (한번 만들면 재사용)

물리 법칙을 실시간으로 계산하고 3D로 렌더링하는 핵심 엔진. 각 엔진은 독립적이며, 하나의 엔진 위에서 다양한 챌린지가 생성된다.

| 엔진 | 커버 범위 | 개념·변수 상세 |
|------|----------|--------------|
| 투사체 엔진 | 포물선 운동, 중력, 마그누스, 코리올리, 중력보조, 시공간 곡률 | [concepts.md](./contents/engines/projectile/concepts.md) |
| 충돌/에너지 엔진 | 마블런, 충돌, 에너지 보존, 진자, 루브 골드버그, 푸코 진자 | [concepts.md](./contents/engines/collision-energy/concepts.md) |
| 파동 엔진 | 정상파, 간섭, 도플러, 마하콘, 공명, 건축음향, 악기물리 | [concepts.md](./contents/engines/wave/concepts.md) |
| 분자 엔진 | VSEPR, 입체화학, 극성, 반응속도론, 단백질접힘, DNA, 결정구조 | [concepts.md](./contents/engines/molecular/concepts.md) |
| 궤도 엔진 | 케플러, 조석력, 라그랑주, 삼체문제, 쌍성계, 은하충돌 | [concepts.md](./contents/engines/orbit/concepts.md) |
| 양자 엔진 | 이중슬릿, 터널링, 오비탈, 불확정성, 얽힘, 광전효과 | [concepts.md](./contents/engines/quantum/concepts.md) |

#### Layer 2 — 챌린지 (메타데이터 기반 동적 생성)

엔진 하나에서 변수 조합을 바꾸는 것만으로 무한한 챌린지가 생성된다. 각 챌린지는 `tags`, `difficulty`, `relatedConcepts` 등의 **메타데이터**를 가지며, 적응형 엔진이 이 메타데이터를 기반으로 런타임에 다음 챌린지를 동적으로 선택한다. FSM(상태 기계)처럼 다음 챌린지를 하드코딩하지 않고, 메타데이터 위의 규칙 엔진이 사용자 상태에 맞는 최적의 챌린지를 골라낸다.

예시 — 투사체 엔진 하나로:
- "지구에서 45도로 던져봐"
- "달에서 같은 조건으로?"
- "공기저항 켜봐"
- "질량 2배로 하면?"
- "목성에서는?"

챌린지의 본질은 **"엔진 + 변수 조합(`params`) + 예측 질문 + 메타데이터(`tags`, `difficulty`, `relatedConcepts`)"라는 데이터(JSON)**이다. 이 구조가 AI 자동 생성, 적응형 경로, UGC를 동시에 가능하게 한다.

#### Layer 3 — 공간 테마 + Discover 콘텐츠 (수동 제작, 게임 스테이지 품질)

각 실험실이 **게임 스테이지처럼 느껴지는 것**이 게이미피케이션의 핵심이다. 공간 테마는 비주얼(Skybox, 조명, 색상 팔레트, 머티리얼, 파티클 이펙트), 사운드(앰비언트 사운드, BGM), 연출(진입 전환 애니메이션, 리워드 이펙트)을 포함한다.

**Discover 콘텐츠는 concept-level 라이브러리로 구조화한다.** 개념별 설명을 한 번만 작성하고, 챌린지가 `relatedConcepts`로 해당 개념을 참조한다. 각 개념 설명은 3단계 깊이를 가진다:

| 깊이 | 설명 | 예시 (중력) |
|------|------|-----------|
| **Level 1** (직관적 비유) | 일상 경험에 비유한 한 줄 설명 | "지구가 공을 잡아당기는 힘" |
| **Level 2** (개념 설명) | 변수 간 관계를 자연어로 설명 | "질량이 클수록, 가까울수록 강해진다" |
| **Level 3** (수식 포함) | 정량적 관계를 수식으로 표현 | F = GMm/r² |

정답/오답에 따른 프레이밍(축하, 오류 설명)은 스키마에 하드코딩하지 않고 **런타임 로직**으로 처리한다. 챌린지 JSON의 `discover` 필드에는 `relatedConcepts`와 선택적 `contextHint`만 포함하고, 사용자의 정답/오답 상태 + concept 깊이 레벨에 맞춰 런타임에 설명을 조합한다. LLM 초안 + 감수로 확장 가능.

### 3.2 크로스 참조 시스템 — 엔진 간 지식 그래프

엔진 간에 공유되는 물리 개념을 **크로스 참조 맵**으로 연결한다. 챌린지의 `relatedConcepts`가 이 지식 그래프의 노드를 참조하며, 사용자가 한 엔진에서 개념을 학습한 후 다른 엔진의 관련 챌린지를 추천하는 **크로스 엔진 추천**의 데이터 기반이 된다.

| 개념 | 주 엔진 (정의) | 참조 엔진 (사용) | 연결 맥락 |
|------|---------------|-----------------|----------|
| 중력 | 투사체 (표면 중력) | 궤도 (만유인력) | "지구에서의 g=9.8은 만유인력의 특수한 경우" |
| 적색편이 | 파동 (도플러 효과) | 궤도 (은하 후퇴 속도) | "파동에서 배운 도플러가 우주 팽창의 증거가 된다" |
| 전자기 스펙트럼 | 파동 (파장/진동수) | 양자 (광전효과, 흑체복사) | "파동의 에너지가 양자화되어 있다" |
| 에너지 양자화 | 양자 (플랑크) | 분자 (전자 에너지 준위) | "원자의 전자가 불연속 에너지 준위에 있는 이유" |
| 파동 간섭 | 파동 (물결, 소리) | 양자 (이중슬릿) | "물결 간섭과 같은 원리가 양자 세계에서도 작동" |
| 확률구름/오비탈 | 양자 (파동함수) | 분자 (전자 배치, 결합) | "오비탈의 모양이 화학결합의 방향을 결정" |

이 크로스 참조 맵의 상세 정의는 `docs/contents/engines/README.md`를 원본으로 한다.

### 3.3 4개 공간 — 게임 스테이지

각 실험실은 고유한 공간 테마를 가지며, 실험실을 해금하는 것 자체가 "새로운 스테이지를 여는" 흥분을 제공한다.

| 공간 | 트랙 | 포함 엔진 |
|------|------|----------|
| 🔬 역학 실험실 | 고전물리 | 투사체, 충돌/에너지, 파동 |
| 🧪 분자 실험실 | 화학 | 분자 |
| 🔭 우주 관측소 | 우주과학 | 궤도 |
| ⚛️ 양자 연구소 | 양자역학 | 양자 |

각 공간의 비주얼 테마, 사운드, 전환 연출 상세는 [UX Design — Space Themes](./ux-design.md)를 참조한다.

### 3.4 확장 모델: 가로 × 세로 × 깊이

콘텐츠는 세 방향으로 확장된다. 모든 방향이 엔진 기반이라 확장 비용이 점진적으로 줄어든다.

```
가로 확장 — 새로운 공간이 열린다
  "역학 실험실을 클리어했더니 우주 관측소가 해금되었다!"
  비용: 새 공간 테마(Skybox/조명/파티클/머티리얼/BGM/앰비언트 사운드/전환 연출) + 새 엔진

세로 확장 — 기존 공간이 넓어진다
  "역학 실험실에 전자기 스테이션이 새로 생겼다!"
  비용: 새 엔진만 (공간 테마는 기존 것 재사용)

깊이 확장 — 기존 엔진에 챌린지가 추가된다
  "투사체 스테이션에 새로운 챌린지 팩이 왔다!"
  비용: 변수 조합 / 파라미터 세트만 (매우 낮음, AI 자동 생성 가능)
```

이 구조에서는 "24개 모듈을 수동 제작"할 필요가 없다. 엔진을 하나 만들면 그 위에서 챌린지가 무한히 생성되고, 공간을 하나 만들면 그 안에서 스테이션이 계속 추가된다.

### 3.5 Phase Expansion Tree

```
Phase 1 (Launch)
└── 🔬 역학 실험실
    ├── 투사체 스테이션 (투사체 엔진)
    ├── 에너지 스테이션 (충돌/에너지 엔진)
    └── 파동 스테이션 (파동 엔진)

Phase 2
└── 🔬 역학 실험실

    ├── 파동 스테이션
    ├── 소리/빛 스테이션 ← 세로 확장 (파동 엔진 재사용 + 새 변수)
    └── 전자기 스테이션 ← 세로 확장 (새 엔진)

Phase 3

Phase 4

Phase 5
├── 🔬 역학 실험실
├── 🧪 분자 실험실
├── 🔭 우주 관측소
└── ⚛️ 양자 연구소 ← 가로 확장 (새 공간 + 양자 엔진)
```

## 4. AI System: 적응형 경험 엔진

풀 AI 챗봇이 아니라, 게임의 **디렉터 AI**에 가까운 역할이다.

### AI의 4가지 역할

| 역할 | 설명 | 입력 | 출력 |
|------|------|------|------|
| **챌린지 선택** | 챌린지 풀에서 메타데이터(`tags`, `difficulty`, `relatedConcepts`)를 기반으로 사용자 수준에 맞는 다음 챌린지를 동적으로 선택 | 사용자 예측 이력, 태그별 정확도, 현재 difficulty | 다음 챌린지 ID |
| **난이도 조절** | 연속 정답/오답에 따라 `difficulty` 필터 범위를 조정하여 적절한 난이도의 챌린지를 선택 | 연속 정답/오답 수, 반응 시간 | difficulty 필터 범위 |
| **약점 보강** | `tags`별 정확도를 추적하여 낮은 정확도 태그의 챌린지를 우선 배치 | 태그별 정확도 히스토리 | 우선 태그 목록 |
| **설명 깊이 조절** | Discover 단계에서 사용자 수준에 맞는 concept-level 설명 깊이(level1/2/3) 선택 | 사용자 레벨, 이전 이해도 | 설명 깊이 레벨 |

### Phase별 AI 구현 전략

| Phase | 접근 방식 | 설명 |
|-------|----------|------|
| **Phase 1** | 메타데이터 기반 규칙 엔진 | 챌린지의 `tags`/`difficulty`/`relatedConcepts` 메타데이터 위에서 규칙 기반으로 다음 챌린지를 선택. FSM(상태 기계) 방식의 하드코딩된 분기가 아닌, 메타데이터 필터링 + 우선순위 규칙 |
| **Phase 2+** | ML 모델 전환 | Phase 1에서 쌓인 메타데이터 + 사용자 행동 데이터로 학습, 개인화 정확도 향상 |

## 5. XR Strategy: 체감 증폭기

**Must 3D/XR — 모든 챌린지는 3D에서만 의미 있어야 하고, XR이 체험을 증폭해야 한다.**

콘텐츠 설계 원칙: 2D에서는 불가능하거나 무의미한 챌린지만 만든다. 3D가 "있으면 좋은 것"이 아니라 "없으면 안 되는 것"이어야 한다. XR 없이도 100% 동작하되, XR 전환 시 체험이 질적으로 증폭되어야 한다.

XR은 God Hand 모델의 모든 단계를 신체적 경험으로 증폭한다:

| 코어 루프 단계 | XR 증폭 효과 |
|-------------|------------|
| **Predict** | 실제로 손으로 궤적을 그리며 예측. 신체적 예측이 인지적 참여를 높인다 |
| **Play** | 공간에서 직접 던지고, 분자를 손으로 조립. "내 손의 감각"이 개념 이해에 연결된다 |
| **Discover** | 예측한 궤적과 실제 궤적이 3D 공간에서 겹쳐 보임. 틀린 지점이 공간적으로 드러난다 |
| **환경 전환** | 방 전체의 환경(조명, 배경, 사운드스케이프)이 실험실 테마에 맞게 바뀌며 게임 스테이지 몰입감을 극대화 |

### Progressive XR 전략

- **콘텐츠 설계 (Phase 1~):** Must 3D/XR — 모든 챌린지가 3D에서만 의미 있어야 하며, XR 증폭 시나리오를 설계에 포함. 각 챌린지의 XR 증폭은 [challenges.md](./contents/engines/README.md) 참조
- **Phase 1 구현:** 3D 웹. WebXR 대응 아키텍처 확보 (God Hand 멘탈 모델 + 동일 코드베이스)
- **Phase 2+:** XR 모드 구현. 동일 씬에서 입력만 핸드 트래킹으로 전환

## 6. Wedge Strategy

### 왜 역학 실험실(고전물리)부터인가

1. **최대 사용자 풀** — 초등부터 성인까지 모든 연령이 경험하는 가장 기본적인 과학 영역
2. **직관적 인터랙션** — 투사체, 롤러코스터 → 게이미피케이션에 최적화
3. **코어 루프 최적 적용** — "예측 → 실험 → 발견" 루프를 가장 자연스럽게 적용 가능
4. **God Hand 검증** — 핵심 인터랙션(집기, 던지기, 그리기)을 검증하기에 최적
5. **풍부한 변수 공간** — 엔진 3개(투사체, 충돌/에너지, 파동)로 다양한 챌린지 확보
6. **외부 의존 없음** — 외부 데이터 없이 빠른 출시 가능
7. **기술 기반 검증** — 엔진 아키텍처, 코어 루프, 적응형 분기 등 공통 기술 기반 검증

### 왜 양자 연구소가 마지막인가

양자 연구소(이중슬릿 실험의 "관측기 ON/OFF")는 **바이럴 잠재력과 게이미피케이션 적합도가 동시에 가장 높다.** 역학 실험실로 기술과 코어 루프를 검증한 뒤, 양자 연구소를 바이럴 + 게임 경험의 정점으로 활용한다.

### 공간 해금 순서 근거

```
역학 실험실 (Phase 1) — 기술 검증 + 최대 TAM
    ↓
분자 실험실 (Phase 3) — 화학으로 과목 확장
    ↓
우주 관측소 (Phase 4) — NASA API 연동 + 대중적 매력
    ↓
양자 연구소 (Phase 5) — 바이럴 정점 + 성인 학습자 확장
```

## 7. UGC Vision (유저 생성 콘텐츠)

엔진 기반 구조의 궁극적 도달점은 **유저가 직접 챌린지, 스테이션, 공간을 만드는 것**이다. 스타크래프트의 유즈맵이 엔진 하나로 타워디펜스, RPG, AOS라는 전혀 다른 게임을 만들어낸 것처럼, PhysPlay의 시뮬레이션 엔진 위에서 유저가 자신만의 과학 실험을 설계하고 공유하는 생태계를 만든다.

이것이 가능한 이유는 챌린지의 본질이 "엔진 + 변수 조합 + 예측 질문"이라는 데이터이기 때문이다. AI가 변수 조합을 선택해서 챌린지를 만드는 것과, 유저가 변수 조합을 선택해서 챌린지를 만드는 것은 구조적으로 동일하다.

### UGC의 3단계

| 단계 | 설명 | 사용 시나리오 | 타이밍 |
|------|------|-------------|--------|
| **1. 챌린지 제작/공유** | 유저가 엔진의 변수를 세팅하고 "이거 풀어봐"라고 URL로 공유 | 친구에게 도전장을 보내고, 커뮤니티에서 공유 | Phase 3 |
| **2. 스테이션/미션 제작** | 챌린지를 엮어 시퀀스로 구성 | "목성 탈출 미션" 같은 커스텀 스테이션을 기존 공간 안에 추가 | Phase 4+ |
| **3. 공간 제작** | 테마 프리셋(Skybox, 조명, 파티클)을 조합하고 스테이션을 배치 | "유체역학 풀장", "소리의 동굴" 같은 커뮤니티 공간 | Phase 5+ |

### UGC가 해결하는 문제들

- **콘텐츠 지속성** — PhysPlay 팀은 엔진만 만들고, 콘텐츠는 커뮤니티가 만든다
- **바이럴** — "내가 만든 챌린지" 공유가 자연스러운 유입 경로
- **가로/세로 확장** — 유저가 직접 공간을 추가하고, 기존 공간을 확장

### UGC를 위한 기술 전제

Phase 1부터 챌린지/스테이션/공간을 **코드가 아닌 데이터(JSON)**로 정의한다. 엔진은 이 데이터를 읽어서 실행하는 구조로, 나중에 에디터 UI만 붙이면 UGC가 열린다.

## 8. Phase 1 Scope: 역학 실험실

### 8.1 What needs to be true

역학 실험실 안에 3개 스테이션이 **"예측 → 실험 → 발견" 코어 루프 + God Hand 인터랙션 + 엔진 기반 챌린지 생성**으로 동작해야 한다.

### 8.2 투사체 스테이션

**엔진:** 투사체 엔진 — 10개 챌린지
**예측 유형:** trajectory 70%, placement 20%, pattern 10%
**God Hand 패턴:** 던지기/발사, 배치, 경로 설계

→ 챌린지 시나리오 상세: [projectile/challenges.md](./contents/engines/projectile/challenges.md)
→ 개념·변수: [projectile/concepts.md](./contents/engines/projectile/concepts.md)

### 8.3 에너지 스테이션

**엔진:** 충돌/에너지 엔진 — 8개 챌린지
**예측 유형:** binary ~40%, pattern ~40%, placement ~10%
**God Hand 패턴:** 조립/빌딩, 던지기, 당기기

→ 챌린지 시나리오 상세: [collision-energy/challenges.md](./contents/engines/collision-energy/challenges.md)
→ 개념·변수: [collision-energy/concepts.md](./contents/engines/collision-energy/concepts.md)

### 8.4 파동 스테이션

**엔진:** 파동 엔진 — 8개 챌린지
**예측 유형:** placement 50%, pattern 37.5%, binary 12.5%
**God Hand 패턴:** 배치/설치, 공간 이동, 당기기

→ 챌린지 시나리오 상세: [wave/challenges.md](./contents/engines/wave/challenges.md)
→ 개념·변수: [wave/concepts.md](./contents/engines/wave/concepts.md)

### 8.5 성능 요구사항

- PC: mid-range 데스크톱/랩톱에서 **60fps** 유지 (기본 타깃)
- 모바일: mid-range 모바일 디바이스에서 **30fps 이상** 유지 (핵심 경험 보장)
- 물리 시뮬레이션 정확도: 교육적 정확성 수준 (연구 도구 수준이 아님)
- 초기 로딩: 3초 이내

### 8.6 챌린지 생성 요구사항

- 각 엔진에서 변수 조합으로 **최소 8개 이상의 챌린지** 제공 (상세 시나리오는 `docs/contents/engines/*/challenges.md` 참조)
- 챌린지 데이터는 JSON으로 정의 — `tags`, `difficulty`, `relatedConcepts` 메타데이터 포함 (UGC 대비)
- 메타데이터 기반 규칙 엔진으로 적응형 챌린지 선택 (Phase 1)

## 9. Phased Roadmap

| Phase | Duration | Key Activities | 확장 방향 |
|-------|----------|---------------|----------|
| **Discovery** | 2-3주 | 투사체 엔진 프로토타입 + God Hand + 코어 루프 통합 | — |
| **Phase 1 (역학 실험실)** | 2-3개월 | 엔진 3개(투사체, 충돌/에너지, 파동) + 역학 실험실 공간 + 규칙 기반 적응형 분기 + 챌린지 생성 시스템 + 랜딩 + i18n + PostHog 트래킹 + 출시 | 첫 공간 |
| **Phase 2 (역학 확장 + AI)** | 1-2개월 | 역학 실험실에 소리/빛, 전자기 스테이션 추가 + 적응형 AI v1 + 피드백 반영 | 세로 확장 |
| **Phase 3 (분자 실험실)** | 3-4개월 | 분자 엔진 + 분자 실험실 공간 + 계정/진도 시스템 + 챌린지 에디터(UGC 1단계) | 가로 확장 |
| **Phase 4 (우주 관측소)** | 3-4개월 | 궤도 엔진 + 우주 관측소 공간 + NASA API 연동 + 스테이션/미션 제작(UGC 2단계) | 가로 확장 |
| **Phase 5 (양자 연구소)** | 3-4개월 | 양자 엔진 + 양자 연구소 공간 + 바이럴 마케팅 + 공간 제작(UGC 3단계) | 가로 확장 |
| **Phase 6+** | Ongoing | 기존 공간 세로 확장 + 깊이 확장 + 신규 공간 (생명과학, 유체역학 등) | 모든 방향 |

## 10. Analytics & Metrics

### Event Tracking Plan

| 이벤트 | 설명 | Properties |
|--------|------|-----------|
| `session_start` | 세션 시작 | device_type (pc/mobile/xr), referrer |

| `play_start` | 실험 시작 | station_id, challenge_id |
| `play_complete` | 실험 완료 | station_id, challenge_id, duration_ms |
| `discover_view` | 발견 화면 진입 | station_id, challenge_id, was_correct |
| `discover_detail_open` | 상세 설명 열기 | station_id, challenge_id, detail_type |
| `challenge_complete` | 챌린지 완료 | station_id, challenge_id, was_correct, attempt_count |
| `station_complete` | 스테이션 완주 | station_id, space_id, total_challenges, correct_count |
| `station_navigate` | 스테이션 간 이동 | from_station_id, to_station_id |
| `space_unlock` | 공간 해금 | space_id |
| `xr_mode_enter` | XR 모드 진입 | device_type |
| `onboarding_complete` | 온보딩(첫 챌린지) 완료 | duration_ms, was_correct |

### 핵심 분석 뷰

| 뷰 | 목적 | 구성 이벤트 |
|----|------|-----------|
| **코어 루프 퍼널** | Predict→Play→Discover 단계별 이탈률 | predict_submit → play_start → discover_view |
| **예측 참여 대시보드** | 예측 참여율 & 스킵율 모니터링 | predict_submit, predict_skip |
| **학습 효과 추적** | 스테이션 내 예측 정확도 변화 | challenge_complete (was_correct 시계열) |
| **탐험 패턴** | 스테이션/공간 간 이동 흐름 | station_navigate, space_unlock |

## 11. Non-functional Requirements

| 카테고리 | 요구사항 |
|---------|---------|
| **Performance** | PC mid-range에서 60fps, 모바일 mid-range에서 30fps 이상, 초기 로딩 3초 이내 |
| **i18n** | 모든 사용자 대면 텍스트는 en, ko 지원 |
| **Responsive** | PC 퍼스트 (마우스/키보드 기본). 반응형 레이아웃과 터치 제스처 매핑으로 모바일 핵심 경험 유지 |
| **Dark mode** | Light/Dark 테마 지원 |
| **Accessibility** | 키보드 내비게이션, 스크린 리더 지원 (2D UI), 색약 대응 컬러 팔레트 |
| **Offline** | Phase 1은 로컬 저장 (IndexedDB), 오프라인 시뮬레이션 실행 가능 |
| **Browser support** | Chrome, Safari, Firefox 최신 2개 버전. WebGPU → WebGL fallback |
| **보안** | 사용자 데이터 최소 수집, Phase 1은 계정 없음 (로컬 저장) |

## 14. Assumptions, Risks & Dependencies

핵심 베팅 4개는 [Product Brief — Key Bets](./product-brief.md#3-key-bets)를 참조. 아래는 전체 가정/리스크 목록.

| Assumption / Risk | Type | Impact | How We'll Address It |
|-------------------|------|--------|---------------------|
| "예측→실험→발견" 루프가 자유 탐색보다 학습 효과가 높다 | Value | High | A/B 테스트: 자유 탐색 vs 예측 루프 모드, 사전/사후 이해도 비교 (n=20) |
| 초등학생부터 성인까지 동일 코어 루프로 커버할 수 있다 | Value | High | 연령대별 사용자 테스트 — 예측 참여율·완주율·재방문율 비교 |
| 예측 참여가 학습 동기(다음 모듈 진행)를 실제로 높인다 | Value | High | Phase 1에서 모듈 완주율 측정 (목표: 40%) |
| 3D 인터랙션이 2D 다이어그램보다 개념 이해도를 높인다 | Value | High | 사용자 테스트에서 사전/사후 이해도 비교 |
| 엔진 기반 변수 조합 챌린지가 수동 제작 챌린지만큼 교육 효과가 있다 | Value | Medium | Phase 1에서 수동 설계 챌린지 vs 변수 조합 챌린지 비교 |
| 적응형 난이도 AI가 1인 개발로 구현 가능한 수준이다 | Feasibility | Medium | Phase 1은 규칙 기반 분기로 시작, AI 모델은 Phase 2+에서 도입 |
| 브라우저에서 물리 시뮬레이션을 60fps로 구동할 수 있다 | Feasibility | High | 물체 던지기 + 롤러코스터 프로토타입 eng spike (2주) |
| God Hand 테이블탑 모델이 웹↔XR 전환에서 일관된 경험을 제공한다 | Feasibility | Medium | Discovery에서 물체 던지기 프로토타입으로 웹/XR 양쪽 테스트 |
| 3D/XR 동일 코드베이스로 Progressive XR 구조를 유지할 수 있다 | Feasibility | Medium | XR 모드 전환 프로토타입 검증 (Discovery) |
| 무료 웹 접근만으로 유의미한 트래픽을 확보할 수 있다 | Viability | High | 출시 후 1개월 내 10K UV 달성 여부로 판단 |
| 초등 저학년 사용자의 텍스트 읽기 부담 | Usability | Medium | 비주얼/애니메이션 중심 가이드 + 적응형 설명 깊이 조절로 대응 |
| XR 브라우저 지원 수준이 브라우저별로 다름 | Dependency | Medium | 3D First — XR은 보너스, 3D가 기본 |
| PDB, NASA API 등 외부 데이터 소스의 안정성과 라이선스 | Dependency | Low | Phase 2+ 진입 전 라이선스 검토, 캐싱/폴백 구현 |
