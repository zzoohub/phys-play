## 1. Phase Goal

**"예측 -> 실험 -> 발견" 코어 루프가 과학 학습 동기를 만든다는 가설을, 역학 실험실 3개 스테이션으로 증명한다.**

Phase 1이 끝났을 때, 사용자가 URL을 열어 30초 안에 첫 챌린지를 체험하고, 3개 스테이션(투사체, 에너지, 파동)에서 예측-실험-발견 루프를 반복하며, "다음 챌린지가 궁금해서 계속하는" 경험이 동작해야 한다. 실험실에 진입하면 풀스크린 3D + HUD로 몰입이 끊기지 않고, 네온 톤 실험 놀이터의 게임 스테이지 분위기가 "여기서 더 놀고 싶다"는 감각을 만들어야 한다.

---

## 2. Scoping Rationale

### 왜 이 범위인가

**최소 증명 단위 = 코어 루프 x 3가지 변주.** 코어 루프의 가치를 검증하려면 한 가지 예측 방식만으로는 부족하다. 투사체(궤적 그리기), 에너지(이진 선택), 파동(패턴 선택)의 세 가지 예측 유형이 모두 동작해야 "예측이라는 메카닉 자체"가 검증된다.

**기술 기반 검증.** 3개 엔진(투사체, 충돌/에너지, 파동)은 물리 시뮬레이션의 핵심 카테고리를 커버한다. 이 엔진들이 PC 브라우저에서 60fps로 동작하면, 이후 Phase의 모든 엔진(분자, 궤도, 양자)도 같은 아키텍처에서 구현 가능하다는 확신을 얻는다.

**풀스크린 3D + HUD 검증.** 실험실 내 모든 UI를 HUD 오버레이로 처리하는 구조가 몰입과 사용성을 동시에 잡는지 검증한다. 이 구조가 Phase 1에서 증명되면, 이후 모든 실험실에 동일 패턴을 적용할 수 있다.

**게임 스테이지 분위기 검증.** 역학 실험실의 비주얼 테마(네온 톤, 형광 격자, 파티클), 사운드(BGM, 앰비언트), 연출(진입 전환, 리워드 이펙트)이 "교육 도구가 아니라 게임"이라는 첫인상을 만드는지 검증한다.

**계정 없이 출시.** Phase 1의 목표는 "와우, 이거 재밌다"라는 반응을 만드는 것이다. 회원가입 장벽은 바이럴과 첫 경험을 죽인다. 진도는 IndexedDB 로컬 저장으로 충분하다.

**Phase 2를 위한 데이터 수집.** 예측 참여율, 정확도 변화, 스테이션 간 이동 패턴 데이터를 수집하여, Phase 2의 ML 기반 적응형 AI 학습 데이터로 활용한다.

### 왜 지금 이것을 빼는가

- **계정/인증**: 로컬 저장으로 시작. 계정은 B2B(교사 대시보드)가 필요한 Phase 3에서 도입
- **XR 모드**: God Hand 멘탈 모델은 유지하되, 실제 핸드 트래킹 구현은 Phase 2+. 아키텍처만 XR-ready로 설계
- **ML 기반 AI**: 데이터가 없는 상태에서 ML은 의미 없음. Phase 1의 규칙 기반 분기 데이터로 Phase 2에서 전환
- **UGC 에디터**: 데이터 구조(JSON)는 Phase 1부터 UGC-ready로 설계하지만, 에디터 UI는 Phase 3+
- **B2B 기능**: 교사 대시보드, 학급 관리, 커스텀 챌린지 구성은 Phase 3+

## 3. Phase Requirements

### 3.1 Core Loop (P0)

코어 루프가 동작하지 않으면 제품이 존재하지 않는다.

| Priority | ID | Requirement | Rationale |
|----------|-----|-------------|-----------|
| P0 | REQ-100 | 사용자는 챌린지마다 결과를 예측한 후 실험을 수행할 수 있다. 예측 없이 실험만 하는 것은 불가하다 (예측 스킵은 명시적 행위로 추적) | 예측이 코어 루프의 시작점. 예측 없는 실험은 PhET과 다를 바 없다 |
| P0 | REQ-101 | 예측은 3가지 유형을 지원한다: 궤적 그리기(trajectory), 이진 선택(binary), 패턴 선택(pattern) | 3개 스테이션이 각각 다른 예측 유형을 사용 |
| P0 | REQ-102 | 실험 실행 후 예측과 결과가 시각적으로 비교된다 (궤적 오버레이, 정답/오답 표시, 패턴 비교) | "내 예측 vs 실제"의 시각적 대비가 인지적 갈등의 핵심 |
| P0 | REQ-103 | 예측이 틀렸을 때: 왜 틀렸는지 개념 설명 + 관련 변수를 조절할 수 있는 기회 제공 | 틀림 -> 호기심 -> 학습의 경로 |
| P0 | REQ-104 | 예측이 맞았을 때: 축하 피드백 + 새로운 변수를 추가한 더 어려운 챌린지로 분기 | 맞음 -> 자신감 -> 도전의 경로 |
| P0 | REQ-105 | 챌린지 완료 후 다음 챌린지가 자동으로 제시된다 (규칙 기반 적응형 분기) | 루프의 연속성. "다음이 궁금해서 계속하는" 동기 |

### 3.10 풀스크린 3D + HUD (P0)

실험실 진입 시 화면 전체가 3D 뷰포트가 되고, 모든 UI는 3D 위에 떠 있는 HUD(Head-Up Display) 오버레이로 표시되어야 한다. 코어 루프(예측 -> 실험 -> 발견) 전체가 3D 환경 안에서 완결되며, 실험 중 2D 페이지로 빠지지 않는다.

| Priority | ID | Requirement | Rationale |
|----------|-----|-------------|-----------|
| P0 | REQ-1050 | 실험실(스테이션) 진입 시 화면 전체가 3D 뷰포트로 전환된다. 실험실 내에서는 별도의 2D 페이지로 이동하지 않는다 | 풀스크린 3D 몰입. 코어 루프 전체가 3D 안에서 완결 |
| P0 | REQ-1051 | Predict 단계의 예측 프롬프트(질문 텍스트, 입력 UI, 제출 버튼)는 3D 뷰포트 위의 HUD 오버레이로 표시된다 | Predict 단계에서도 3D 환경이 유지되어 시뮬레이션 대상을 관찰하면서 예측 가능 |
| P0 | REQ-1052 | Discover 단계의 설명 패널(개념 설명, 비교 시각화, 다음 챌린지 버튼)은 3D 뷰포트 위의 HUD 오버레이로 표시된다 | Discover 단계에서도 시뮬레이션 결과가 3D로 보이는 상태에서 설명을 확인 |
| P0 | REQ-1053 | 변수 조절 컨트롤(슬라이더, 토글 등)은 HUD 오버레이로 표시된다. 변수 변경 시 3D 시뮬레이션에 실시간 반영된다 | 변수 조작과 3D 결과 관찰이 동시에 일어나야 직관적 이해가 가능 |
| P0 | REQ-1054 | HUD 오버레이는 3D 시뮬레이션의 핵심 영역(물체 운동 궤적, 충돌 지점, 파동 간섭 패턴 등)을 가리지 않도록 배치되어야 한다 | HUD가 시뮬레이션 가시성을 방해하면 학습 효과와 몰입이 동시에 저하 |
| P1 | REQ-1055 | HUD 오버레이는 반투명이며, 사용자가 HUD 영역 외의 3D 공간을 클릭/터치하면 HUD가 최소화(또는 축소)되어 3D 뷰를 방해하지 않는다 | 사용자가 시뮬레이션에 집중하고 싶을 때 HUD를 일시적으로 치울 수 있어야 함 |
| P1 | REQ-1056 | 스테이션 내 챌린지 선택/이동도 HUD를 통해 이루어진다. 허브로 돌아가지 않고 스테이션 내에서 챌린지를 탐색할 수 있다 | 3D를 벗어나지 않는 탐색. 몰입 유지 |
| P1 | REQ-1057 | HUD 레이아웃은 PC(넓은 화면)와 모바일(좁은 화면)에서 각각 최적화된 배치를 제공한다. PC에서는 좌/우 사이드 패널 배치, 모바일에서는 하단 시트 또는 최소화 가능한 카드 배치 | 디바이스별 화면 비율에 맞는 HUD 배치. PC에서 3D 영역 중앙을 최대한 확보 |

### 3.11 역학 실험실 분위기 — 게임 스테이지 (P0)

Phase 1의 역학 실험실은 교육 도구가 아니라 **게임 스테이지**로 느껴져야 한다. 진입하는 순간 "여기는 실험실이 아니라 놀이터다"라는 첫인상이 게이미피케이션의 출발점이다.

## 5. Phase User Journeys

### Journey 1: 첫 방문 (First Visit)

> 민준이 친구가 보낸 링크를 클릭한다.

1. URL(`/play`) 접속 -> 로딩 화면 (3초 이내)
2. **역학 실험실 진입 전환 연출** — 화면이 3D 공간으로 전환되며 네온 조명이 켜지고, 형광 격자 바닥이 활성화되고, 경쾌한 일렉트로닉 BGM이 시작된다. 파티클이 부유하는 밝은 실험 놀이터가 펼쳐진다
3. 곧바로 투사체 스테이션의 첫 챌린지가 풀스크린 3D 뷰포트에서 열린다 (회원가입/선택 화면 없음)
4. **PREDICT (HUD)** — 3D 뷰포트 위에 HUD 오버레이로 질문 표시: "이 공을 던지면 어디에 떨어질까? 궤적을 그려보세요"
   - 마우스 클릭-드래그 힌트 애니메이션이 HUD에 표시 (모바일에서는 터치 드로잉 가이드)
   - 민준이 마우스로 예상 궤적을 그린다 (모바일: 손가락으로 드로잉)
   - HUD의 제출 버튼 클릭 (또는 Enter 키)
5. **PLAY** — 공이 God Hand 시점에서 날아간다. 던지는 순간 충격 효과음 재생
   - 예측 궤적(반투명 네온 라인)과 실제 궤적(밝은 실선)이 3D 공간에 동시에 표시
   - 착지 시 바닥 격자에 충격 이펙트
6. **DISCOVER (HUD)** — 예측이 틀림. 3D 뷰포트 위에 HUD 패널로 설명 표시
   - "중력이 지구(9.8m/s2)에서 공을 아래로 당기기 때문에 더 빨리 떨어졌어요"
   - 시뮬레이션 결과가 3D로 유지된 채, HUD 패널에서 설명을 읽을 수 있음
   - "다시 해볼래요? 이번에는 달에서!" 버튼 (HUD)
7. 다음 챌린지 자동 진입 (달 환경으로 변수 변경) — 3D 환경이 미세하게 변화 (배경 변경)하지만 실험실은 유지
8. 2-3개 챌린지 후 **실험실에서 허브로 전환** — 3D 씬이 줌아웃되며 2D 연구소 허브로 전환
9. 허브에서 3개 스테이션(투사체/에너지/파동) 확인. 투사체에 진행률이 표시됨
10. 에너지 스테이션을 클릭하여 진입 -> 다시 3D 전환 연출과 함께 풀스크린 실험실로

**Phase 1 한계:** 브라우저를 닫으면 IndexedDB에 진도 저장. 다른 기기에서는 진도 미동기화.

### Journey 2: 재방문 (Return Visit)

> 민준이 다음 날 PC에서 다시 접속한다.

1. URL(`/play`) 접속 -> 연구소 허브(2D)가 홈 화면으로 표시
2. 이전 진도가 IndexedDB에서 복원됨 (투사체 60%, 에너지 미시작, 파동 미시작)
3. 투사체 스테이션 클릭 -> 진입 전환 연출 -> 풀스크린 3D 역학 실험실 (네온 조명, BGM 재개)
4. 마지막 진행 지점부터 재개. HUD에 현재 챌린지 정보 표시
5. 챌린지 계속 진행 (마우스로 예측 그리기, 키보드 Spacebar로 시뮬레이션 실행)
6. 투사체 스테이션 완주 -> 네온 파티클 폭발 + 실험실 전체 빛나는 리워드 연출 + 축하 효과음
7. HUD에 "에너지 스테이션도 도전해 보세요" 추천 표시
8. 허브로 돌아가서 에너지 스테이션 시작

**Phase 1 한계:** 브라우저 데이터 삭제 시 진도 초기화. 안내 메시지 표시.

### Journey 3: 에너지 스테이션 챌린지 (Energy Challenge Flow)

> 민준이 에너지 스테이션에 처음 진입한다.

1. 허브에서 에너지 스테이션 클릭 -> 진입 전환 연출 -> 풀스크린 3D
2. 롤러코스터 트랙이 3D로 표시. 네온 조명 아래 트랙이 빛나고, 카트가 출발 지점에 대기
3. **PREDICT (HUD)** — "이 높이에서 출발하면 카트가 루프를 통과할 수 있을까?" -> HUD에 예/아니오 버튼
4. **PLAY** — 카트 발사. 에너지 바(HUD 오버레이)가 실시간으로 변함 (위치 에너지 -> 운동 에너지). 카트가 트랙을 따라 이동하는 소리와 충돌음
5. **DISCOVER (HUD)** — 예측이 맞음
   - HUD 패널: "맞았어요! 위치 에너지가 운동 에너지로 바뀌면서 카트가 루프를 통과했습니다"
   - 클리어 리워드 이펙트 (네온 파티클 버스트)
   - "마찰을 추가하면 어떻게 될까요?" (HUD)
6. 다음 챌린지: 마찰 계수 추가. 같은 높이에서 카트 발사
7. **PREDICT (HUD)** -> 예/아니오 -> **PLAY** -> 카트가 루프 중간에 멈춤 (마찰으로 감속하는 효과음)
8. **DISCOVER (HUD)** — "마찰이 에너지를 열로 변환해서, 루프를 통과할 에너지가 부족해졌어요"

### Journey 4: 파동 스테이션 챌린지 (Wave Challenge Flow)

> 민준이 파동 스테이션에 진입한다.

1. 허브에서 파동 스테이션 클릭 -> 진입 전환 연출 -> 풀스크린 3D
2. 물 표면(또는 3D 파동 시뮬레이션)이 표시. 파원 위치에 빛나는 마커
3. **PREDICT (HUD)** — "파원 2개에서 파동이 나오면 어떤 패턴이 생길까?" -> HUD에 3개 패턴 선택지 (썸네일 카드)
4. **PLAY** — 시뮬레이션 실행. 간섭 패턴이 실시간으로 형성. 파동 전파 시 물결 소리 효과음
5. **DISCOVER (HUD)** — 예측이 틀림
   - 선택한 패턴 vs 실제 패턴 비교 (HUD 패널에서 나란히 표시, 3D 시뮬레이션은 유지)
   - "두 파동이 만나면 보강 간섭(높아짐)과 상쇄 간섭(사라짐)이 동시에 일어나요"
   - 3D 뷰에서 보강/상쇄 간섭 영역이 네온 색상으로 하이라이트
6. 다음 챌린지: "파원을 3개로 늘리면?"
7. 더 복잡한 패턴 선택지가 HUD에 제공됨

### 6.2 규칙 기반 적응형 분기 (Phase 1 AI 대체)

Phase 1에서는 ML 모델 대신 규칙 기반 분기를 사용한다.

**분기 규칙:**

| 조건 | 행동 |
|------|------|
| 현재 챌린지 정답 | `branching.on_correct`로 이동 (난이도 상승 또는 새 변수 추가) |
| 현재 챌린지 오답 | `branching.on_incorrect`로 이동 (같거나 낮은 난이도의 관련 챌린지) |
| 연속 2회 오답 | 힌트 표시 후 재시도 기회 제공 |
| 연속 3회 오답 | 더 기초적인 챌린지로 분기 + 추가 개념 설명 |
| 연속 3회 정답 | 난이도 2단계 점프 (중간 챌린지 스킵) |
| 스테이션 내 모든 핵심 챌린지 완료 | 스테이션 완주 처리. 허브 복귀 + 다음 스테이션 추천 |

**난이도 설계 원칙:**

```
Level 1: 단일 변수, 기본 환경 (지구, 기본값)
Level 2: 환경 변경 (달, 목성)
Level 3: 새 변수 추가 (공기저항 ON, 마찰 ON)
Level 4: 복합 변수 (환경 + 추가 변수)
Level 5: 반직관적 시나리오 (질량이 달라도 공기저항 없으면 같은 궤적)
```

| 코어 루프 작동 | 예측 참여율 (스테이션 진입자 중 1회 이상 예측) | 70% | 80% | Event: predict_submit / station_enter |

| 탐험 행동 | 스테이션 간 이동률 (2개 이상 스테이션 방문) | 30% | 50% | Event: station_navigate |

| 정성 피드백 | 교사/교육자 인터뷰 완료 | 10명 | -- | Interview records |

**Counter-metrics:**

| Counter-metric | 위험 신호 | 모니터링 방법 |
|----------------|----------|--------------|
| PC 성능 저하 | mid-range PC에서 60fps 미달 | 실디바이스 테스트 + performance monitoring |
| 모바일 성능 저하 | mid-range 모바일에서 30fps 미달 | 실디바이스 테스트 + performance monitoring |
| 예측 스킵율 과다 | 30% 이상이 예측을 건너뜀 | Event: predict_skip |
| 챌린지 반복감 | "같은 걸 또 하는 느낌" 피드백 | 정성 피드백 (인터뷰, 설문) |
| 온보딩 이탈 | 첫 챌린지 완료율 60% 미만 | Event: onboarding_complete / session_start |
| Discover 스킵 | Discover 단계를 바로 닫는 비율 50% 이상 | Event: discover_view duration |
| HUD가 시뮬레이션 가림 | "UI가 시뮬레이션을 가려서 보기 어렵다" 피드백 | 정성 피드백 (인터뷰, 설문) + discover_view 중 HUD 최소화 빈도 |

| `reward_effect_shown` | 클리어 리워드 이펙트 표시 | `station_id`, `challenge_id`, `reward_type` (challenge_clear / station_complete) | P1 |

| `hint_shown` | 힌트 표시 | `station_id`, `challenge_id`, `consecutive_failures` | P1 |
| `branch_triggered` | 적응형 분기 발생 | `station_id`, `from_challenge_id`, `to_challenge_id`, `branch_reason` | P1 |
| `sound_setting_change` | 사운드 설정 변경 | `setting_type` (bgm/sfx/ambient/mute), `value` | P2 |
| `locale_change` | 언어 변경 | `from_locale`, `to_locale` | P2 |
| `theme_change` | 테마 변경 | `from_theme`, `to_theme` | P2 |
| `error` | 에러 발생 | `error_type`, `error_message`, `station_id`, `challenge_id` | P0 |

### 8.2 Key Funnels

**온보딩 퍼널:**
```
session_start -> lab_transition_complete -> onboarding_start -> predict_submit (first) -> play_complete (first) -> onboarding_complete -> hub_view
```

**코어 루프 퍼널 (반복):**
```
challenge_start -> predict_submit -> play_start -> play_complete -> discover_view -> challenge_complete
```

**탐험 퍼널:**
```
station_complete -> station_navigate -> station_enter (다른 스테이션)
```

## 10. Phase Risks

| Risk | Type | Impact | Likelihood | Mitigation |
|------|------|--------|------------|------------|
| PC 브라우저에서 풀스크린 3D + 물리 시뮬레이션 + HUD 오버레이 60fps 미달 | Feasibility | High | Medium | Engine Spike (Week 1-2)에서 조기 검증. 풀스크린 3D + HUD 구조를 포함한 종합 퍼포먼스 테스트. 물리 연산 최적화 또는 시각적 복잡도 조절 |
| 궤적 그리기 마우스 UX가 부정확하거나 번거로움 | Usability | High | Medium | 프로토타입 단계에서 5명 이상 PC 사용자 테스트. 마우스 드로잉의 부드러움, 클릭-드래그 반응성 검증. 오차 허용 범위 조절 또는 간소화된 입력 대안(예: 핀 포인트 방식) 준비 |
| HUD 오버레이가 3D 시뮬레이션 가시성을 방해 | Usability | High | Medium | HUD 레이아웃을 Predict/Play/Discover 단계별로 최적화 (Play 시에는 HUD 최소화). 사용자 테스트에서 "시뮬레이션이 안 보인다" 피드백 모니터링. 투명도/위치 조절 옵션 준비 |
| 예측 단계가 "귀찮은 절차"로 느껴져 스킵 과다 | Value | High | Medium | 예측 스킵율 30% 초과 시 UX 개선 (예측을 더 재밌게, 또는 예측 난이도 조절). A/B 테스트 준비 |
| 게임 스테이지 에셋(비주얼 테마, 파티클, BGM, 효과음, 전환 연출) 제작 비용/시간 초과 | Feasibility | High | Medium | Atmosphere Alpha (Week 4-5)에서 역학 실험실의 핵심 에셋만 1차 적용. 에셋 품질은 "인상적이지만 과하지 않은" 수준으로 스코핑. 무료/저가 사운드/파티클 에셋 라이브러리 활용. 부족 시 파티클과 BGM을 P1으로 격하 |
| 변수 조합 챌린지가 반복감을 줌 | Value | Medium | Medium | 환경 테마 변화(지구/달/목성)로 시각적 다양성 확보. 반직관적 시나리오(Level 5)로 놀라움 유지. 게임 스테이지 분위기(파티클, 사운드)가 변수 변경 시 미세하게 변화하여 신선함 제공 |
| 3초 이내 로딩 미달성 | Feasibility | Medium | Low | 코드 스플리팅, 에셋 레이지 로딩, 3D 씬 및 사운드 파일 경량화 |
| 규칙 기반 분기가 너무 기계적으로 느껴짐 | Usability | Medium | Medium | 분기 규칙을 충분히 다양화. 같은 정답/오답이라도 컨텍스트에 따라 다른 경로 |
| 모바일에서 30fps 미달 또는 HUD 사용성 저하 | Feasibility | Medium | Medium | 모바일은 P0이지만 PC 이후 우선순위. 자동 품질 조절(REQ-1204)로 파티클/그림자 축소. 모바일 전용 HUD 레이아웃(REQ-1057) 별도 검증 |
| 교사 이메일 50건 미수집 | Viability | Low | Medium | 교사 커뮤니티(인디스쿨 등) 타겟 마케팅. 제품 자체의 교육 가치로 입소문 |
| IndexedDB 데이터 유실 (브라우저 정리, 시크릿 모드) | Usability | Low | High | 명시적 안내 + Phase 3 계정 전환 시 마이그레이션 경로 설계 |

## 11. Open Questions (Phase 1 Specific)

- [ ] **궤적 오차 판정 기준**: 착지점 +-15%로 시작하되, PC 마우스 드로잉과 모바일 터치 드로잉에서 각각 "너무 엄격/관대" 피드백에 따라 조절. 마우스가 터치보다 정밀하므로 디바이스별 오차 허용 범위를 다르게 가져갈 것인가?
- [ ] **적응형 분기의 세부 파라미터**: 연속 정답 3회 -> 2단계 점프가 적절한가? 사용자 테스트 필요
- [ ] **Discover HUD 패널의 크기와 위치**: 3D 시뮬레이션 결과를 가리지 않으면서 충분한 설명을 보여줄 수 있는 최적 크기는? PC(사이드 패널)와 모바일(하단 시트)의 최적 비율은?
- [ ] **HUD 투명도와 배경 블러**: HUD 오버레이의 기본 투명도를 몇 %로 설정할 것인가? 배경 블러를 적용하면 가독성은 높아지지만 3D가 흐려짐 — 밸런스 기준은?
- [ ] **파동 시각화 방식**: 2D top-down vs 3D 물결 표면. 성능과 직관성 트레이드오프. Engine Spike에서 검증 필요
- [ ] **에너지 바/물리량 시각화 디자인**: 에너지 전환을 "보여주는" 최적의 HUD 시각화 방식은? 바 차트, 색상 그라데이션, 파티클 등
- [ ] **역학 실험실 BGM 선택**: 경쾌한 일렉트로닉이라는 방향은 정했지만, 구체적 BPM, 분위기(chiptune? synthwave? lo-fi electronic?)는? 레퍼런스 게임의 BGM 벤치마크 필요
- [ ] **효과음 제작 vs 라이브러리**: 물리 상호작용 효과음(던지기, 충돌, 파동 전파)을 자체 제작할 것인가, 사운드 라이브러리에서 조달할 것인가? 예산과 품질 트레이드오프
- [ ] **진입 전환 연출 소요 시간**: 허브 -> 실험실 전환 연출의 목표 소요 시간은? 너무 길면 반복 진입 시 짜증, 너무 짧으면 스테이지 진입감 부족. 1-2초가 적절한가?
- [ ] **스킵 가능한 전환 연출**: 전환 연출을 첫 진입에서만 풀로 보여주고, 재진입 시에는 스킵/단축할 수 있게 할 것인가?
- [ ] **챌린지 10개/스테이션 충분한가?**: 완주까지 평균 소요 시간과 사용자 만족도에 따라 조절. 너무 적으면 빈약, 너무 많으면 피로
- [ ] **온보딩 첫 챌린지 선택**: 투사체 스테이션의 "지구에서 45도 던지기"가 PC 마우스로 가장 직관적인가? 다른 시나리오가 더 나을 수 있음
- [x] **~~랜딩 페이지 vs 바로 챌린지~~**: 결정됨 — `/`는 랜딩(SEO/설명), `/play`는 바로 챌린지(바이럴/공유). REQ-1700, REQ-1700a 참조
- [ ] **WebGPU vs WebGL 시각적 차이**: fallback 시 어떤 시각 효과를 포기할 것인가? 역학 실험실의 네온 파티클과 형광 격자가 가장 영향 받을 가능성
- [ ] **IndexedDB 저장 스키마**: 최소 필수 필드 정의 필요. 서버 마이그레이션을 고려한 설계
- [ ] **모바일 품질 자동 조절 기준**: 어떤 성능 지표(fps, GPU 사용률)를 모니터링하여 품질 단계를 자동 전환할 것인가?
