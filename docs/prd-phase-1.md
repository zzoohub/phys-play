## 1. Phase Goal

**"예측 -> 실험 -> 발견" 코어 루프가 과학 학습 동기를 만든다는 가설을, 역학 실험실 3개 스테이션으로 증명한다.**

Phase 1이 끝났을 때, 사용자가 URL을 열어 30초 안에 첫 챌린지를 체험하고, 3개 스테이션(투사체, 에너지, 파동)에서 예측-실험-발견 루프를 반복하며, "다음 챌린지가 궁금해서 계속하는" 경험이 동작해야 한다.

---

## 2. Scoping Rationale

### 왜 이 범위인가

**최소 증명 단위 = 코어 루프 x 3가지 변주.** 코어 루프의 가치를 검증하려면 한 가지 예측 방식만으로는 부족하다. 투사체(궤적 그리기), 에너지(이진 선택), 파동(패턴 선택)의 세 가지 예측 유형이 모두 동작해야 "예측이라는 메카닉 자체"가 검증된다.

**기술 기반 검증.** 3개 엔진(투사체, 충돌/에너지, 파동)은 물리 시뮬레이션의 핵심 카테고리를 커버한다. 이 엔진들이 브라우저에서 60fps로 동작하면, 이후 Phase의 모든 엔진(분자, 궤도, 양자)도 같은 아키텍처에서 구현 가능하다는 확신을 얻는다.

**계정 없이 출시.** Phase 1의 목표는 "와우, 이거 재밌다"라는 반응을 만드는 것이다. 회원가입 장벽은 바이럴과 첫 경험을 죽인다. 진도는 IndexedDB 로컬 저장으로 충분하다.

**Phase 2를 위한 데이터 수집.** 예측 참여율, 정확도 변화, 스테이션 간 이동 패턴 데이터를 수집하여, Phase 2의 ML 기반 적응형 AI 학습 데이터로 활용한다.

### 왜 지금 이것을 빼는가

- **계정/인증**: 로컬 저장으로 시작. 계정은 B2B(교사 대시보드)가 필요한 Phase 3에서 도입
- **XR 모드**: God Hand 멘탈 모델은 유지하되, 실제 핸드 트래킹 구현은 Phase 2+. 아키텍처만 XR-ready로 설계
- **ML 기반 AI**: 데이터가 없는 상태에서 ML은 의미 없음. Phase 1의 규칙 기반 분기 데이터로 Phase 2에서 전환
- **UGC 에디터**: 데이터 구조(JSON)는 Phase 1부터 UGC-ready로 설계하지만, 에디터 UI는 Phase 3+
- **B2B 기능**: 교사 대시보드, 학급 관리, 커스텀 챌린지 구성은 Phase 3+

## 3. Phase Requirements

### 3.1 Core Loop (P0)

코어 루프가 동작하지 않으면 제품이 존재하지 않는다.

| Priority | ID | Requirement | Rationale |
|----------|-----|-------------|-----------|
| P0 | REQ-100 | 사용자는 챌린지마다 결과를 예측한 후 실험을 수행할 수 있다. 예측 없이 실험만 하는 것은 불가하다 (예측 스킵은 명시적 행위로 추적) | 예측이 코어 루프의 시작점. 예측 없는 실험은 PhET과 다를 바 없다 |
| P0 | REQ-101 | 예측은 3가지 유형을 지원한다: 궤적 그리기(trajectory), 이진 선택(binary), 패턴 선택(pattern) | 3개 스테이션이 각각 다른 예측 유형을 사용 |
| P0 | REQ-102 | 실험 실행 후 예측과 결과가 시각적으로 비교된다 (궤적 오버레이, 정답/오답 표시, 패턴 비교) | "내 예측 vs 실제"의 시각적 대비가 인지적 갈등의 핵심 |
| P0 | REQ-103 | 예측이 틀렸을 때: 왜 틀렸는지 개념 설명 + 관련 변수를 조절할 수 있는 기회 제공 | 틀림 -> 호기심 -> 학습의 경로 |
| P0 | REQ-104 | 예측이 맞았을 때: 축하 피드백 + 새로운 변수를 추가한 더 어려운 챌린지로 분기 | 맞음 -> 자신감 -> 도전의 경로 |
| P0 | REQ-105 | 챌린지 완료 후 다음 챌린지가 자동으로 제시된다 (규칙 기반 적응형 분기) | 루프의 연속성. "다음이 궁금해서 계속하는" 동기 |

## 5. Phase User Journeys

### Journey 1: 첫 방문 (First Visit)

> 민준이 친구가 보낸 링크를 클릭한다.

1. URL 접속 -> 로딩 화면 (3초 이내)
2. 곧바로 투사체 스테이션의 첫 챌린지가 열린다 (회원가입/선택 화면 없음)
3. **PREDICT** — 화면 안내: "이 공을 던지면 어디에 떨어질까? 궤적을 그려보세요"
   - 민준이 손가락으로 예상 궤적을 그린다
   - 궤적 제출 버튼 탭
4. **PLAY** — 공이 God Hand 시점에서 날아간다
   - 예측 궤적(반투명)과 실제 궤적(실선)이 동시에 표시
5. **DISCOVER** — 예측이 틀림
   - "중력이 지구(9.8m/s2)에서 공을 아래로 당기기 때문에 더 빨리 떨어졌어요"
   - "다시 해볼래요? 이번에는 달에서!" 버튼
6. 다음 챌린지 자동 진입 (달 환경으로 변수 변경)
7. 2-3개 챌린지 후 연구소 허브가 열림
8. 허브에서 3개 스테이션(투사체/에너지/파동) 확인. 투사체에 진행률이 표시됨
9. 에너지 스테이션을 탭하여 새로운 스테이션 시작

**Phase 1 한계:** 브라우저를 닫으면 IndexedDB에 진도 저장. 다른 기기에서는 진도 미동기화.

### Journey 2: 재방문 (Return Visit)

> 민준이 다음 날 다시 접속한다.

1. URL 접속 -> 연구소 허브가 홈 화면으로 표시
2. 이전 진도가 IndexedDB에서 복원됨 (투사체 60%, 에너지 미시작, 파동 미시작)
3. 투사체 스테이션 탭 -> 마지막 진행 지점부터 재개
4. 챌린지 계속 진행
5. 투사체 스테이션 완주 -> 축하 메시지 + "에너지 스테이션도 도전해 보세요" 추천
6. 허브로 돌아가서 에너지 스테이션 시작

**Phase 1 한계:** 브라우저 데이터 삭제 시 진도 초기화. 안내 메시지 표시.

### Journey 3: 에너지 스테이션 챌린지 (Energy Challenge Flow)

> 민준이 에너지 스테이션에 처음 진입한다.

1. 스테이션 진입 -> 롤러코스터 트랙이 3D로 표시
2. **PREDICT** — "이 높이에서 출발하면 카트가 루프를 통과할 수 있을까?" -> 예/아니오 선택
3. **PLAY** — 카트 발사. 에너지 바가 실시간으로 변함 (위치 에너지 -> 운동 에너지)
4. **DISCOVER** — 예측이 맞음
   - "맞았어요! 위치 에너지가 운동 에너지로 바뀌면서 카트가 루프를 통과했습니다"
   - "마찰을 추가하면 어떻게 될까요?"
5. 다음 챌린지: 마찰 계수 추가. 같은 높이에서 카트 발사
6. **PREDICT** -> 예/아니오 -> **PLAY** -> 카트가 루프 중간에 멈춤
7. **DISCOVER** — "마찰이 에너지를 열로 변환해서, 루프를 통과할 에너지가 부족해졌어요"

### Journey 4: 파동 스테이션 챌린지 (Wave Challenge Flow)

> 민준이 파동 스테이션에 진입한다.

1. 스테이션 진입 -> 물 표면(또는 2D 파동 시뮬레이션)이 표시
2. **PREDICT** — "파원 2개에서 파동이 나오면 어떤 패턴이 생길까?" -> 3개 패턴 중 선택
3. **PLAY** — 시뮬레이션 실행. 간섭 패턴이 실시간으로 형성
4. **DISCOVER** — 예측이 틀림
   - 선택한 패턴 vs 실제 패턴 비교
   - "두 파동이 만나면 보강 간섭(높아짐)과 상쇄 간섭(사라짐)이 동시에 일어나요"
   - 보강/상쇄 간섭 영역이 색상으로 하이라이트
5. 다음 챌린지: "파원을 3개로 늘리면?"
6. 더 복잡한 패턴 선택지가 제공됨

### 6.2 규칙 기반 적응형 분기 (Phase 1 AI 대체)

Phase 1에서는 ML 모델 대신 규칙 기반 분기를 사용한다.

**분기 규칙:**

| 조건 | 행동 |
|------|------|
| 현재 챌린지 정답 | `branching.on_correct`로 이동 (난이도 상승 또는 새 변수 추가) |
| 현재 챌린지 오답 | `branching.on_incorrect`로 이동 (같거나 낮은 난이도의 관련 챌린지) |
| 연속 2회 오답 | 힌트 표시 후 재시도 기회 제공 |
| 연속 3회 오답 | 더 기초적인 챌린지로 분기 + 추가 개념 설명 |
| 연속 3회 정답 | 난이도 2단계 점프 (중간 챌린지 스킵) |
| 스테이션 내 모든 핵심 챌린지 완료 | 스테이션 완주 처리. 허브 복귀 + 다음 스테이션 추천 |

**난이도 설계 원칙:**

```
Level 1: 단일 변수, 기본 환경 (지구, 기본값)
Level 2: 환경 변경 (달, 목성)
Level 3: 새 변수 추가 (공기저항 ON, 마찰 ON)
Level 4: 복합 변수 (환경 + 추가 변수)
Level 5: 반직관적 시나리오 (질량이 달라도 공기저항 없으면 같은 궤적)
```

## 8. Analytics Event Tracking Plan (Phase 1)

### 8.1 Event Catalog

| Event | Trigger | Properties | Priority |
|-------|---------|-----------|----------|
| `session_start` | 앱 로드 완료 | `device_type`, `browser`, `referrer`, `locale`, `is_returning` | P0 |
| `onboarding_start` | 첫 챌린지 진입 | `referrer` | P0 |
| `onboarding_complete` | 첫 챌린지 완료 | `duration_ms`, `was_correct` | P0 |
| `hub_view` | 허브 화면 진입 | `unlocked_stations`, `completed_stations` | P0 |
| `station_enter` | 스테이션 진입 | `station_id`, `space_id`, `entry_point` (hub / recommendation / onboarding) | P0 |
| `challenge_start` | 챌린지 시작 | `station_id`, `challenge_id`, `difficulty` | P0 |
| `predict_submit` | 예측 제출 | `station_id`, `challenge_id`, `predict_type`, `predict_value`, `time_to_predict_ms` | P0 |
| `predict_skip` | 예측 스킵 | `station_id`, `challenge_id` | P0 |
| `play_start` | 시뮬레이션 시작 | `station_id`, `challenge_id` | P0 |
| `play_complete` | 시뮬레이션 종료 | `station_id`, `challenge_id`, `duration_ms` | P0 |
| `discover_view` | Discover 화면 진입 | `station_id`, `challenge_id`, `was_correct` | P0 |
| `discover_detail_open` | 상세 설명 펼침 | `station_id`, `challenge_id`, `detail_type` | P1 |
| `discover_dismiss` | Discover 화면 닫기 | `station_id`, `challenge_id`, `duration_ms` | P0 |
| `challenge_complete` | 챌린지 완료 (Discover 이후 다음으로 진행) | `station_id`, `challenge_id`, `was_correct`, `attempt_count`, `difficulty` | P0 |
| `station_complete` | 스테이션 완주 | `station_id`, `space_id`, `total_challenges`, `correct_count`, `duration_ms` | P0 |
| `station_navigate` | 스테이션 간 이동 | `from_station_id`, `to_station_id`, `trigger` (hub / recommendation) | P0 |
| `hint_shown` | 힌트 표시 | `station_id`, `challenge_id`, `consecutive_failures` | P1 |
| `branch_triggered` | 적응형 분기 발생 | `station_id`, `from_challenge_id`, `to_challenge_id`, `branch_reason` | P1 |
| `locale_change` | 언어 변경 | `from_locale`, `to_locale` | P2 |
| `theme_change` | 테마 변경 | `from_theme`, `to_theme` | P2 |
| `error` | 에러 발생 | `error_type`, `error_message`, `station_id`, `challenge_id` | P0 |

### 8.2 Key Funnels

**온보딩 퍼널:**
```
session_start -> onboarding_start -> predict_submit (first) -> play_complete (first) -> onboarding_complete -> hub_view
```

**코어 루프 퍼널 (반복):**
```
challenge_start -> predict_submit -> play_start -> play_complete -> discover_view -> challenge_complete
```

**탐험 퍼널:**
```
station_complete -> station_navigate -> station_enter (다른 스테이션)
```

## 10. Phase Risks

| Risk | Type | Impact | Likelihood | Mitigation |
|------|------|--------|------------|------------|
| 브라우저 물리 시뮬레이션 60fps 미달 | Feasibility | High | Medium | Engine Spike (Week 1-2)에서 조기 검증. 물리 연산 최적화 또는 시각적 복잡도 조절 |
| 궤적 그리기 터치 UX가 부정확하거나 번거로움 | Usability | High | Medium | 프로토타입 단계에서 5명 이상 사용자 테스트. 오차 허용 범위 조절 또는 간소화된 입력 대안 준비 |
| 예측 단계가 "귀찮은 절차"로 느껴져 스킵 과다 | Value | High | Medium | 예측 스킵율 30% 초과 시 UX 개선 (예측을 더 재밌게, 또는 예측 난이도 조절). A/B 테스트 준비 |
| 변수 조합 챌린지가 반복감을 줌 | Value | Medium | Medium | 환경 테마 변화(지구/달/목성)로 시각적 다양성 확보. 반직관적 시나리오(Level 5)로 놀라움 유지 |
| 3초 이내 로딩 미달성 | Feasibility | Medium | Low | 코드 스플리팅, 에셋 레이지 로딩, 3D 씬 경량화 |
| 규칙 기반 분기가 너무 기계적으로 느껴짐 | Usability | Medium | Medium | 분기 규칙을 충분히 다양화. 같은 정답/오답이라도 컨텍스트에 따라 다른 경로 |
| 교사 이메일 50건 미수집 | Viability | Low | Medium | 교사 커뮤니티(인디스쿨 등) 타겟 마케팅. 제품 자체의 교육 가치로 입소문 |
| IndexedDB 데이터 유실 (브라우저 정리, 시크릿 모드) | Usability | Low | High | 명시적 안내 + Phase 3 계정 전환 시 마이그레이션 경로 설계 |

## 11. Open Questions (Phase 1 Specific)

- [ ] **궤적 오차 판정 기준**: 착지점 +-15%로 시작하되, 사용자 테스트에서 "너무 엄격/관대" 피드백에 따라 조절. 최적값은?
- [ ] **적응형 분기의 세부 파라미터**: 연속 정답 3회 -> 2단계 점프가 적절한가? 사용자 테스트 필요
- [ ] **Discover 설명 분량과 깊이**: 모바일에서 긴 텍스트는 읽히지 않음. 이미지/애니메이션 비율은? 설명 길이 상한은?
- [ ] **파동 시각화 방식**: 2D top-down vs 3D 물결 표면. 성능과 직관성 트레이드오프. Engine Spike에서 검증 필요
- [ ] **에너지 바/물리량 시각화 디자인**: 에너지 전환을 "보여주는" 최적의 시각화 방식은? 바 차트, 색상 그라데이션, 파티클 등
- [ ] **챌린지 10개/스테이션 충분한가?**: 완주까지 평균 소요 시간과 사용자 만족도에 따라 조절. 너무 적으면 빈약, 너무 많으면 피로
- [ ] **온보딩 첫 챌린지 선택**: 투사체 스테이션의 "지구에서 45도 던지기"가 가장 직관적인가? 다른 시나리오가 더 나을 수 있음
- [x] **~~랜딩 페이지 vs 바로 챌린지~~**: 결정됨 — `/`는 랜딩(SEO/설명), `/play`는 바로 챌린지(바이럴/공유). REQ-1700, REQ-1700a 참조
- [ ] **WebGPU vs WebGL 시각적 차이**: fallback 시 어떤 시각 효과를 포기할 것인가? 파동 스테이션이 가장 영향 받을 가능성
- [ ] **IndexedDB 저장 스키마**: 최소 필수 필드 정의 필요. 서버 마이그레이션을 고려한 설계
