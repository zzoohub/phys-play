## 1. Problem

### What problem are we solving?

과학 교육에는 세 겹의 문제가 겹쳐 있다.

**(1) 차원의 문제 — 3D 개념을 2D 매체로 전달하는 한계**
분자 구조, 전자 오비탈, 파동 간섭, 궤도 역학 등 본질적으로 3차원인 개념이 2D 교과서와 정적 다이어그램으로 전달된다. 학생은 머릿속에서 3D를 재구성해야 하며, 이 인지 부담이 이해를 가로막는다.

**(2) 순서의 문제 — 수식이 직관보다 먼저 온다**
수식이 직관보다 먼저 제시되는 역순 학습 구조에서, 수학에 약한 학생은 과학 자체를 포기한다. "F=ma를 외웠지만 힘과 가속도의 관계가 어떤 느낌인지 모른다"는 것이 전형적인 증상이다.

**(3) 수동성의 문제 — "만질 수 있다"와 "배운다"는 다르다**
PhET처럼 인터랙티브 시뮬레이션이 있어도, 슬라이더를 만지작거리다 나가는 것이 현실이다. 능동적 사고를 유도하는 구조가 없으면 "만지는 교육"도 "보는 교육"과 다르지 않다. 학습 과학 연구에 따르면, 진짜 개념 변화(conceptual change)는 **"내 예측이 틀렸다"는 인지적 갈등(cognitive conflict)** 순간에 발생한다. 현재 어떤 도구도 이 순간을 체계적으로 설계하지 않는다.

**문제의 크기:**
- OECD PISA 2022 기준, 한국 학생의 과학 흥미도는 OECD 평균 이하. 한국 중고등학생 약 270만 명이 과학 교과를 수강하지만, 과학 진로 선택 비율은 지속 하락 추세
- 양자컴퓨팅, 생체분자, 우주산업 등 과학 리터러시 수요가 급증하지만, 기초 개념에 대한 직관적 이해 도구가 부재
- 기존 도구는 2D 위주(PhET), 고가 설치형($50K+/학교, Labster), 수동적 시청(YouTube), 문제풀이 중심(Brilliant, Khan Academy)
- 3D 체험 + 능동적 사고 유도 + 학습 동기 엔진을 통합한 도구가 존재하지 않음

현재 어떤 도구도 **"예측하게 만들고 → 3D로 실험하게 하고 → 틀린 지점에서 개념을 발견하게 하는"** 구조를 제공하지 않는다.

### Why now?

- **브라우저 3D 기술 성숙:** WebGPU 표준화, Three.js/R3F 생태계 성숙으로 브라우저에서 고품질 3D 렌더링과 실시간 물리 시뮬레이션이 실용적 수준에 도달
- **WebXR 표준화:** 별도 앱 없이 브라우저에서 XR 체험 가능. 동일 코드베이스로 3D → XR 점진적 전환
- **XR 기기 보급 확대:** Meta Quest 3, Apple Vision Pro 등 소비자 기기 확산
- **AI 적응형 학습의 실현 가능성:** LLM과 경량 ML 모델의 발전으로, 사용자 수준에 맞는 적응형 난이도 조절이 기술적으로 가능해짐
- **게이미피케이션 교육의 검증:** Duolingo가 게이미피케이션으로 언어 교육 시장을 재정의한 것처럼, 과학 교육에도 같은 기회가 존재. 그러나 아직 과학 분야에서 이를 3D 체험과 결합한 사례가 없음
- **오픈 과학 데이터:** PDB, PubChem, NASA Open API 등 콘텐츠 기반이 되는 공개 데이터가 풍부

## 3. Proposed Direction

### High-level approach

과학을 **"예측하고, 실험하고, 발견하는" 게임**으로 만든다. 학생은 매 챌린지에서 먼저 결과를 예측하고, 3D 시뮬레이션으로 직접 실험한 뒤, 예측과 결과를 비교하며 과학 개념을 스스로 발견한다. 틀린 예측이 다음 단계의 호기심이 되고, 호기심이 학습 동기가 되는 **자기강화 루프(self-reinforcing loop)**를 만든다.

이 경험을 극대화하기 위해 **3D First, XR Enhanced** 구조를 채택하고, 인터랙션 모델로 **"신의 손(God Hand)" 테이블탑 패턴**을 사용한다. 사용자는 1인칭 실험자로서 내 앞에 놓인 실험 대상을 직접 손으로 조작한다. 디바이스 우선순위는 **모바일 퍼스트** (하지만 웹 풀스크린도 당연히 지원해야한)다. Primary 타깃(한국 중고등학생)이 거의 100% 스마트폰으로 접속하기 때문이다. 터치 제스처가 기본 인터랙션이 되며(스와이프로 던지기, 손가락 드로잉, 핀치 줌), 데스크톱(마우스)과 XR(핸드 트래킹)은 같은 엔진에서 입력만 매핑한다. 동일한 멘탈 모델("나 → 내 앞의 대상 → 손으로 조작")이 모든 디바이스에서 유지된다. AI는 사용자의 예측 패턴과 정확도를 분석하여 난이도와 분기를 자동 조절하는 **적응형 엔진** 역할을 한다.

### Core insight

기존 과학 교육 도구의 핵심 문제는 "3D가 아니라서"도 아니고 "만질 수 없어서"도 아니다. **"생각하지 않고 만지기 때문"**이다.

PhET은 인터랙티브하지만 "이 슬라이더를 왜 움직여야 하는지" 동기가 없다. Brilliant는 문제를 풀게 하지만 3D 체험이 없다. YouTube는 시각화가 뛰어나지만 수동적이다. 학습 과학에서 가장 강력한 개념 변화(conceptual change)는 **"내 예측이 틀렸다"는 인지적 갈등** 순간에 발생한다.

PhysPlay의 핵심 베팅은 이것이다: **"예측 → 3D 실험 → 발견"이라는 루프를 게임의 코어 메카닉으로 만들면, 학습 동기와 학습 효과를 동시에 해결할 수 있다.**

두 번째 베팅은 **통합 커리큘럼**이다. 과학은 분야별로 나뉘어 있지만 본질적으로 연결되어 있다. "달에서 공을 던졌더니 예상보다 멀리 갔네?" → "그러면 목성에서는?" → "중력이 빛도 휘게 한다면?" → 고전물리에서 양자역학까지 호기심의 사슬이 자연스럽게 연결되는 경험을 제공한다.

### Core Loop: 예측 → 실험 → 발견

```
┌─────────────────────────────────────────────┐
│                                             │
│   🎯 PREDICT                                │
│   "이 공을 달에서 던지면 어디에 떨어질까?"      │
│   → 사용자가 궤적을 그리거나 선택지를 고른다    │
│                                             │
│           ↓                                 │
│                                             │
│   🧪 PLAY                                   │
│   내 앞의 실험 대상을 직접 손으로 조작한다      │
│   웹: 마우스/터치 | XR: 핸드 트래킹            │
│                                             │
│           ↓                                 │
│                                             │
│   💡 DISCOVER                                │
│   예측 vs 결과 비교                           │
│   ✗ 틀렸다면 → "왜?" → 개념 설명 + 변수 조절  │
│   ✓ 맞았다면 → 변수 추가 → 더 어려운 예측     │
│                                             │
│           ↓                                 │
│                                             │
│   🔄 NEXT CHALLENGE                          │
│   AI가 수준에 맞는 다음 챌린지를 제시           │
│   호기심이 다음 단계의 동기가 된다              │
│                                             │
└─────────────────────────────────────────────┘
```

**예측 방식은 챌린지 성격에 따라 달라진다:**
- **궤적 그리기** — 투사체처럼 궤적이 핵심인 챌린지. "내가 그린 선 vs 실제 궤적"이 겹쳐 보이는 순간이 가장 강력한 인지적 갈등. 정답 판정은 오차 범위 기반
- **이진 선택** — 롤러코스터처럼 "될까/안 될까"가 핵심인 챌린지
- **패턴 선택지** — 파동 간섭처럼 여러 패턴 중 고르는 챌린지

챌린지 데이터에 `predict.type`으로 정의되어 UGC에서도 유저가 예측 방식을 선택할 수 있다.

**코어 루프가 학습 동기를 만드는 원리:**
1. **예측은 참여를 강제한다** — 슬라이더를 만지작거리는 것과 달리, "내 예측"이 걸려 있으면 결과에 관심을 갖게 된다
2. **틀림은 호기심을 만든다** — "왜 내 예상과 다르지?"라는 질문이 자연스럽게 개념 학습으로 이어진다
3. **맞음은 자신감을 만든다** — 더 어려운 챌린지에 도전하는 동기가 된다
4. **연결은 탐험을 만든다** — "달에서 맞았으니, 블랙홀 근처에서는?"이 트랙 간 이동의 동기가 된다

### 스크린 구성: 3D 뷰포트 + 2D UI 레이어

모든 화면이 3D일 필요는 없다. 유저 체감은 "3D 과학 게임"이지만, 실제 구성은 **3D 뷰포트 + 2D UI 오버레이** 혼합이다.

- **3D** — Play 단계 (God Hand 조작, 시뮬레이션 실행, 결과 시각화). 제품의 핵심
- **2D 오버레이** — Predict 단계의 질문/입력, Discover 단계의 설명 패널. 3D 씬 위에 뛄다
- **2D** — 연구소 허브(공간/스테이션 선택), 진도, 설정. 3D 씬을 내려 리소스 절약

이 구조가 모바일 퍼스트에서 특히 중요하다. 허브나 설명 화면에서 3D를 계속 돌리면 배터리/발열 문제가 생기기 때문이다.

### 인터랙션 모델: 1인칭 실험자 + 테이블탑 조작 (God Hand)

캐릭터나 아바타 없이, **"나는 실험자이고, 실험 대상은 내 앞에 있다"**는 일관된 프레임을 모든 모듈에서 유지한다. 이를 "신의 손(God Hand)" 또는 테이블탑 모델이라 부른다.

**왜 이 모델인가:**
- **스케일 전환이 자유롭다** — 태양계를 테이블 위에 축소해서 올려놓고 손으로 돌릴 수도, 분자를 눈앞에 확대해서 손으로 조립할 수도 있다. 원자든 은하든, "내 앞에 놓인 것을 관찰하고 조작한다"는 프레임이면 자연스럽다
- **코어 루프와 완벽하게 맞는다** — "이 공을 던지면 어디에 떨어질까?" → 손으로 직접 던진다 → 결과를 관찰한다. 전체 과정이 "내 손 → 내 눈" 안에서 완결된다
- **웹 ↔ XR 전환 비용이 제로다** — 웹에서 마우스로 시뮬레이션을 조작하는 것이 이미 테이블탑 위의 대상을 조작하는 느낌이다. XR로 전환하면 마우스가 내 손으로 바뀌는 것뿐, 멘탈 모델이 동일하다
- **캐릭터 개발 부담이 없다** — 아바타, 커스터마이징, 3인칭 카메라 등의 복잡도를 완전히 제거한다

**디바이스별 경험 매핑:**

| 모듈 | 웹(3D) — 마우스/터치 | XR — 핸드 트래킹 |
|------|---------------------|-----------------|
| 운동과 힘 | 마우스로 공을 던지고 궤적 관찰 | 손으로 직접 공을 집어 던진다 |
| 롤러코스터 | 마우스로 트랙을 그리고 카트 관찰 | 공중에 손가락으로 트랙을 그린다 |
| 태양계 | 화면에서 줌인/아웃 | 테이블 위 미니 태양계를 손으로 확대/축소 |
| 분자 구조 | 마우스 드래그로 원자 결합 | 양손으로 원자를 잡아 결합 |
| 이중슬릿 | 클릭으로 입자 발사 | 손가락으로 입자를 튕겨 발사 |

모든 경우에 **"나(실험자) → 내 앞의 대상 → 손으로 조작"**이라는 프레임이 일관된다.

### 콘텐츠 아키텍처: 엔진 기반 + 공간 확장

기존 "4트랙 × 6모듈 = 24개 고정 콘텐츠"의 교과서식 구조를 폐기하고, **시뮬레이션 엔진 기반의 확장 가능한 구조**를 채택한다.

**콘텐츠의 3개 레이어:**

**Layer 1 — 시뮬레이션 엔진 (한번 만들면 재사용)**
물리 법칙을 실시간으로 계산하고 3D로 렌더링하는 핵심 엔진. 각 엔진은 독립적이며, 하나의 엔진 위에서 다양한 챌린지가 생성된다.

| 엔진 | 커버 범위 | 핵심 변수 |
|------|----------|----------|
| 투사체 엔진 | 포물선 운동, 중력, 공기저항 | 중력(g), 질량, 각도, 공기저항 계수, 발사 속도 |
| 충돌/에너지 엔진 | 롤러코스터, 충돌, 에너지 보존 | 높이, 마찰, 질량, 탄성 계수 |
| 파동 엔진 | 물결, 간섭, 회절, 소리, 빛 | 주파수, 진폭, 파원 수, 매질 |
| 분자 엔진 | 원자 결합, 분자 구조, 반응 | 원소 종류, 결합 유형, 에너지 |
| 궤도 엔진 | 행성 운동, 중력장, 조석력 | 질량, 거리, 초기 속도, 천체 수 |
| 양자 엔진 | 파동함수, 이중슬릿, 터널링 | 슬릿 간격, 관측 여부, 장벽 높이/폭 |

**Layer 2 — 챌린지 (변수 조합으로 자동 생성)**
엔진 하나에서 변수 조합을 바꾸는 것만으로 무한한 챌린지가 생성된다. AI 적응형 엔진이 사용자 수준에 맞는 변수 조합을 선택하여 다음 챌린지를 제시한다.

예: 투사체 엔진 하나로 → "지구에서 45도로 던져봐" → "달에서 같은 조건으로?" → "공기저항 켜봐" → "질량 2배로 하면?" → "목성에서는?"

**Layer 3 — 공간 테마 + Discover 콘텐츠 (수동 제작, 경량화)**
각 공간의 분위기(Skybox, 조명, 파티클)와 Discover 단계의 개념 설명 텍스트. 복잡한 3D 환경이 아니라 조명/색감/파티클 수준으로 경량화한다. Discover 설명은 LLM 초안 + 감수로 확장 가능.

**4개 공간 테마:**

| 공간 | 트랙 | 비주얼 테마 | 포함 엔진 |
|------|------|-----------|----------|
| 🔬 역학 실험실 | 고전물리 | 밝은 조명, 깔끔한 격자 바닥, 중성 톤 | 투사체, 충돌/에너지, 파동 |
| 🧪 분자 실험실 | 화학 | 약간 어두운 조명, 푸른 빛, 미세한 파티클 | 분자 |
| 🔭 우주 관측소 | 우주과학 | 어둡고, 별 배경, 은은한 앰비언트 라이트 | 궤도 |
| ⚛️ 양자 연구소 | 양자역학 | 네온 느낌, 확률구름 같은 배경 파티클 | 양자 |

**확장 모델: 가로 × 세로 × 깊이**

콘텐츠는 세 방향으로 확장된다. 모든 방향이 엔진 기반이라 확장 비용이 점진적으로 줄어든다.

```
가로 확장 — 새로운 공간이 열린다
  "역학 실험실을 클리어했더니 우주 관측소가 해금되었다!"
  비용: 새 공간 테마(Skybox/조명/파티클) + 새 엔진

세로 확장 — 기존 공간이 넓어진다
  "역학 실험실에 전자기 스테이션이 새로 생겼다!"
  비용: 새 엔진만 (공간 테마는 기존 것 재사용)

깊이 확장 — 기존 엔진에 챌린지가 추가된다
  "투사체 스테이션에 새로운 챌린지 팩이 왔다!"
  비용: 변수 조합 / 파라미터 세트만 (매우 낮음, AI 자동 생성 가능)
```

이 구조에서는 "24개 모듈을 수동 제작"할 필요가 없다. 엔진을 하나 만들면 그 위에서 챌린지가 무한히 생성되고, 공간을 하나 만들면 그 안에서 스테이션이 계속 추가된다. Phase별 확장도 자연스럽다:

```
Phase 1 (Launch)
└── 🔬 역학 실험실
    ├── 투사체 스테이션 (투사체 엔진)
    ├── 에너지 스테이션 (충돌/에너지 엔진)
    └── 파동 스테이션 (파동 엔진)

Phase 2
└── 🔬 역학 실험실
    ├── 투사체 스테이션
    ├── 에너지 스테이션
    ├── 파동 스테이션
    ├── 소리/빛 스테이션 ← 세로 확장 (파동 엔진 재사용 + 새 변수)
    └── 전자기 스테이션 ← 세로 확장 (새 엔진)

Phase 3
├── 🔬 역학 실험실 (계속 확장 가능)
└── 🧪 분자 실험실 ← 가로 확장 (새 공간 + 분자 엔진)
    ├── 원자 결합 스테이션
    ├── 분자 구조 스테이션
    └── 화학 반응 스테이션

Phase 4
├── 🔬 역학 실험실
├── 🧪 분자 실험실
└── 🔭 우주 관측소 ← 가로 확장 (새 공간 + 궤도 엔진)

Phase 5
├── 🔬 역학 실험실
├── 🧪 분자 실험실
├── 🔭 우주 관측소
└── ⚛️ 양자 연구소 ← 가로 확장 (새 공간 + 양자 엔진)
```

### AI의 역할: 적응형 경험 엔진

풀 AI 챗봇이 아니라, 게임의 **디렉터 AI**에 가까운 역할이다:

- **챌린지 생성** — 시뮬레이션 엔진의 변수 조합을 선택하여 사용자 수준에 맞는 챌린지를 자동 생성
- **난이도 조절** — 예측 정확도가 높으면 변수를 추가하고, 낮으면 힌트를 제공하거나 더 기초적인 시나리오로 분기
- **학습 경로 분기** — 사용자의 강점/약점 패턴을 분석하여 다음 챌린지를 개인화
- **설명 깊이 조절** — 발견(Discover) 단계에서 사용자 수준에 맞는 설명 제공 (직관적 비유 vs 수식 포함)
- **Phase 1은 규칙 기반** — if/else 분기로 시작, 데이터가 쌓이면 ML 모델로 전환 (Phase 2+)

### XR의 역할: 체감 증폭기

XR은 God Hand 모델의 모든 단계를 신체적 경험으로 증폭한다:
- **Predict** — 실제로 손으로 궤적을 그리며 예측. 신체적 예측이 인지적 참여를 높인다
- **Play** — 공간에서 직접 던지고, 분자를 손으로 조립. "내 손의 감각"이 개념 이해에 연결된다
- **Discover** — 예측한 궤적과 실제 궤적이 3D 공간에서 겹쳐 보임. 틀린 지점이 공간적으로 드러난다
- **공간 전환** — XR에서는 방 전체의 환경(조명, 배경, 사운드스케이프)이 트랙 테마에 맞게 바뀌며 몰입감을 제공한다

3D가 기본이고, XR은 보너스다. XR 없이도 100% 동작해야 한다.

### 첫 30초: 온보딩 = 첫 번째 챌린지

**첫 방문:** URL을 열면 회원가입/트랙 선택 없이 바로 역학 실험실의 "공 던지기" 챌린지로 진입한다. 30초 안에 예측→실험→발견 한 바퀴를 채험한다. 첫 챌린지 완료 후 연구소 허브가 열리며 전체 공간 맵이 등장한다. 튜토리얼이 곧 첫 번째 챌린지다.

**재방문:** 연구소 허브(2D)가 홈이 된다. 해금된 공간들이 보이고, 아직 잠긴 공간은 실루엿으로 표시되어 호기심을 유발한다. 원하는 공간 → 스테이션 → 챌린지를 자유롭게 선택한다.

### Wedge 전략

**왜 역학 실험실(고전물리)부터인가:**
1. 모든 중고등 교육과정에 포함 → 잠재 사용자 풀 최대
2. 투사체, 롤러코스터 → 직관적이고 게이미피케이션에 최적화된 인터랙션
3. "예측 → 실험 → 발견" 루프를 가장 자연스럽게 적용 가능
4. God Hand 모델의 핵심 인터랙션(집기, 던지기, 그리기)을 검증하기에 최적
5. 엔진 3개(투사체, 충돌/에너지, 파동)로 풍부한 챌린지 변수 공간 확보
6. 외부 데이터 의존 없음 → 빠른 출시
7. 엔진 아키텍처, 코어 루프, 적응형 분기 등 공통 기술 기반을 검증

**왜 양자 연구소가 마지막인가:**
양자 연구소(이중슬릿 실험의 "관측기 ON/OFF")는 **바이럴 잠재력과 게이미피케이션 적합도가 동시에 가장 높다.** 역학 실험실로 기술과 코어 루프를 검증한 뒤, 양자 연구소를 바이럴 + 게임 경험의 정점으로 활용한다.

### What differentiates this?

| 차별점 | 설명 |
|--------|------|
| **예측-실험-발견 루프** | "만지는 교육"을 넘어 "생각하는 교육". 인지적 갈등을 핵심 메카닉으로 설계한 유일한 과학 교육 도구 |
| **God Hand 인터랙션** | 1인칭 실험자 + 테이블탑 조작. 웹에서는 마우스, XR에서는 내 손 — 동일한 멘탈 모델로 디바이스 간 전환 비용 제로 |
| **엔진 기반 무한 챌린지** | 시뮬레이션 엔진 × 변수 조합으로 콘텐츠가 자동 생성된다. 24개 모듈 수동 제작이 아닌, 엔진 위의 무한 챌린지 |
| **확장하는 연구소** | 새 공간 해금 + 기존 공간 확장의 이중 성장. "내 연구소가 커진다"는 소유감과 탐험 동기 |
| **적응형 AI 엔진** | 사용자 수준에 맞게 변수 조합을 선택하여 챌린지를 생성하고, 난이도와 설명 깊이를 자동 조절 |
| **3D-native + XR Enhanced** | 처음부터 3D로 설계하고, XR로 체감을 강화. PhET(2D)도, Labster(VR only)도 아닌 점진적 몰입 |
| **제로 장벽 접근** | URL 하나로 시작, 설치 없음, 기본 무료 |

**핵심 포지션:** PhET의 접근성 + Labster의 3D 몰입 + Brilliant의 인터랙션 + **Duolingo의 학습 동기 엔진**을 브라우저에서 무료로 통합한다.

### 장기 비전: 유저 생성 콘텐츠 (UGC)

엔진 기반 구조의 궁극적 도달점은 **유저가 직접 챌린지, 스테이션, 공간을 만드는 것**이다. 스타크래프트의 유즈맵이 엔진 하나로 타워디펜스, RPG, AOS라는 전혀 다른 게임을 만들어낸 것처럼, PhysPlay의 시뮬레이션 엔진 위에서 유저가 자신만의 과학 실험을 설계하고 공유하는 생태계를 만든다.

이것이 가능한 이유는 챌린지의 본질이 "엔진 + 변수 조합 + 예측 질문"이라는 데이터이기 때문이다. AI가 변수 조합을 선택해서 챌린지를 만드는 것과, 유저가 변수 조합을 선택해서 챌린지를 만드는 것은 구조적으로 동일하다.

**UGC의 3단계:**
- **챌린지 제작/공유** — 유저가 엔진의 변수를 세팅하고 "이거 풀어봐"라고 URL로 공유. 교사가 수업용 챌린지를, 학생이 친구에게 도전장을 보낸다
- **스테이션/미션 제작** — 챌린지를 엮어 시퀀스로 구성. "목성 탈출 미션" 같은 커스텀 스테이션을 기존 공간 안에 추가
- **공간 제작** — 테마 프리셋(Skybox, 조명, 파티클)을 조합하고 스테이션을 배치하여 완전히 새로운 공간을 만든다. "유체역학 풀장", "소리의 동굴" 같은 커뮤니티 공간이 생긴다

이 비전을 실현하기 위해 Phase 1부터 챌린지/스테이션/공간을 **코드가 아닌 데이터(JSON)**로 정의한다. 엔진은 이 데이터를 읽어서 실행하는 구조로, 나중에 에디터 UI만 붙이면 UGC가 열린다.

**UGC가 해결하는 문제들:**
- 콘텐츠 지속성 — PhysPlay 팀은 엔진만 만들고, 콘텐츠는 커뮤니티가 만든다
- B2B 가치 — 교사가 수업 목표에 맞는 챌린지/미션을 직접 설계
- 바이럴 — "내가 만든 챌린지" 공유가 자연스러운 유입 경로
- 가로/세로 확장 — 유저가 직접 공간을 추가하고, 기존 공간을 확장

**타이밍:** Phase 1에서 데이터 구조 설계 → Phase 3에서 교사 챌린지 에디터 → Phase 4+에서 전체 UGC 개방

### What this is NOT (Non-Goals)

- 기존 LMS를 대체하는 것이 아니다 — LMS와 병행 사용
- 학점이나 자격증을 부여하는 공인 교육과정이 아니다 — 체험과 이해가 목적
- 전문 연구자를 위한 고정밀 시뮬레이션 도구가 아니다 — 교육적 정확성과 직관이 우선
- 네이티브 앱이 아니다 — 모든 것은 웹 기반, URL 하나로 접근
- AI 튜터/챗봇이 아니다 — AI는 적응형 경험 엔진이지, 대화형 교사가 아니다
- 포인트만 쌓는 얕은 게이미피케이션이 아니다 — XP/배지는 보조 수단이고, 핵심 동기는 인지적 갈등에서 오는 호기심이다
- 3인칭 RPG가 아니다 — 아바타/캐릭터 없이, 1인칭 실험자로서 과학을 직접 손으로 조작하는 경험이다
- 24개 모듈을 수동으로 만드는 콘텐츠 공장이 아니다 — 엔진을 만들고, 챌린지는 변수 조합과 AI로 생성한다

## 5. Audience & Market Context

### Target audience

**Primary — 중고등학생 (수식 기반 학습에 벽을 느끼는 학생)**

> "물리 시간에 F=ma가 나왔을 때, '이 공을 달에서 던지면 어디에 떨어질까?' 같은 질문에 답해보고, 직접 실험으로 확인하고 싶다. 맞으면 뿌듯하고, 틀리면 왜 틀렸는지 궁금해서 계속하게 될 것 같다."

**Secondary — 과학 기초가 필요한 성인 학습자 (테크/양자컴퓨팅 관심층)**

> "양자컴퓨팅 논문을 읽으려면 양자역학 기초가 필요한데, 내 수준에 맞춰서 챌린지를 풀어가며 직관을 잡을 수 있는 도구가 필요하다."

**Secondary — 시각적 교보재가 필요한 교사**

> "'이러면 어떻게 될까?'를 학생들에게 물은 뒤, 각자 예측하고 시뮬레이션으로 확인하는 수업을 하고 싶다."

## 6. Scope & Timeline

### Product scope

**시뮬레이션 엔진 기반의 확장 가능한 과학 게이미피케이션 플랫폼.** 4개 공간(역학 실험실, 분자 실험실, 우주 관측소, 양자 연구소)이 점진적으로 해금되며, 각 공간 안에서 스테이션이 추가되고, 각 스테이션에서 챌린지가 변수 조합으로 생성된다. 인터랙션은 God Hand 모델로 통일, 브라우저에서 XR까지 일관된 경험을 제공한다.

### What needs to be true to test our hypothesis?

**역학 실험실** 안에 3개 스테이션(투사체, 에너지, 파동)이 **"예측 → 실험 → 발견" 코어 루프 + God Hand 인터랙션 + 엔진 기반 챌린지 생성**으로 동작해야 한다. 구체적으로:

1. **투사체 스테이션** — "달에서 이 공을 던지면 어디에 떨어질까?" → 예측 → God Hand로 던짐 → 비교 → 변수 바꿔서 다음 챌린지 (투사체 엔진의 변수: 중력, 각도, 질량, 공기저항)
2. **에너지 스테이션** — "이 트랙에서 카트가 루프를 통과할 수 있을까?" → 예측 → 트랙 설계 → 에너지 보존 발견 (충돌/에너지 엔진의 변수: 높이, 마찰, 질량)
3. **파동 스테이션** — "두 파동이 만나면 어떤 패턴이 생길까?" → 예측 → 시뮬레이션 → 간섭 발견 (파동 엔진의 변수: 주파수, 진폭, 파원 수)

이 세 가지가 60fps로 동작하고, 각 엔진에서 변수 조합으로 최소 10개 이상의 챌린지가 자동 생성되어야 한다. 예측 참여율 70%, 스테이션 완주율 40%.

### Rough timeline

| Phase | Duration | Key Activities | 확장 방향 |
|-------|----------|---------------|----------|
| Discovery | 2-3주 | 투사체 엔진 프로토타입 + God Hand + 코어 루프 통합, 교사 인터뷰 5명 | — |
| Phase 1 (역학 실험실) | 2-3개월 | 엔진 3개(투사체, 충돌/에너지, 파동) + 역학 실험실 공간 + 규칙 기반 적응형 분기 + 챌린지 생성 시스템 + 랜딩 + i18n + 출시 | 첫 공간 |
| Phase 2 (역학 확장 + AI) | 1-2개월 | 역학 실험실에 소리/빛, 전자기 스테이션 추가 + 적응형 AI v1 + 피드백 반영 | 세로 확장 |

## 7. Open Questions

- [ ] "예측 → 실험 → 발견" 루프의 UX가 번거롭게 느껴지지 않으려면 어떻게 설계해야 하는가? (예측 스킵율 30% 미만 목표)
- [ ] 예측 방식은 무엇이 최적인가? (궤적 그리기 vs 선택지 고르기 vs 수치 입력)
- [ ] 변수 조합 챌린지가 "같은 걸 또 하는 느낌"을 주지 않으려면 어떤 변수를 바꿔야 하는가?
- [ ] 적응형 분기의 규칙 기반(Phase 1)과 AI 기반(Phase 2+)의 경계는?
- [ ] 포인트/배지/리더보드 같은 외적 보상 요소는 어느 수준까지 넣을 것인가?
- [ ] 공간 해금 조건은 무엇인가? (특정 스테이션 클리어? 예측 정확도 달성? 시간 기반?)
- [ ] God Hand 모델에서 스케일 전환(확대/축소) UX는 어떻게 설계하는가?
- [ ] B2B 교사 대시보드에서 "학생별 예측 패턴 분석"의 가치는? (Discovery에서 확인)
- [ ] 첫 사용자 유입 채널: Product Hunt? 교사 커뮤니티? SNS 바이럴? SEO?
- [ ] XR 모드의 우선순위: Phase 1에 포함 vs Phase 2+에서 추가?
- [ ] 학습 진도 저장을 위한 계정 시스템 도입 시점 (Phase 1은 IndexedDB 로컬 저장)
